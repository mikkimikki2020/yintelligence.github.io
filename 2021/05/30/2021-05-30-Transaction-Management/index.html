Transaction Management

VUB 4019814ENR Databases

Video + Slide + Extra

Transaction 交易

  • A transaction is a sequence of operations that form a single unit of work

    事务是形成单个工作单元的一系列操作

  • A transaction is often initiated by an application program

    事务通常由应用程序发起

Example: Ariane Peters wants to transfer 700 Euro from her bank account (314-229) to her landlord's account (889-752)

示例：Ariane Peters 想将 700 欧元从她的银行账户 (314-229) 转入她房东的账户 (889-752)

  • begin a transaction

    START TRANSACTION

  • end a transaction

    COMMIT (if successful) or ROLLBACK (if errors)

    A transaction Ti transforms one consistent database state into another consistent database state

事务 Ti 将一种一致的数据库状态转换为另一种一致的数据库状态

  • during the execution of Ti the DB may be temporarily inconsistent

    在Ti的执行过程中，DB可能会暂时不一致

Either the whole transaction must succeed or the effect of all operations has to be undone (rollback)

要么整个事务必须成功，要么必须撤消所有操作的效果（回滚）

Problem

There are two main transaction issues

  1. concurrent execution of multiple transactions

    多个事务的并发执行

  2. recovery after hardware failures and system crashes

    硬件故障和系统崩溃后的恢复

In many SQL implementations, each SQL statement is a transaction on its own

在许多 SQL 实现中，每个 SQL 语句本身就是一个事务transaction

  • this default behaviour can be disabled

    可以禁用此默认行为

  • SQL:1999 introduced BEGIN ATOMIC ... END blocks

To preserve the integrity of data, the DBMS has to ensure that the so-called ACID properties are fulfilled for any transaction

为了保持数据的完整性，DBMS 必须确保任何事务都满足所谓的ACID 属性

ACID Properties

  • Atomicity 原子性

    • either all operations of a transaction are reflected in the database or none of them (all or nothing)

      要么事务的所有操作都反映在数据库中，要么都不反映（全部或全部）

  • Consistency 一致性

    • if the database was is a consistent state before the transaction started, it will be in a consistent state after the transaction has been executed

      如果数据库在事务开始之前是一致状态，那么在事务执行之后它就会处于一致状态

  • Isolation 隔离性

    • if transactions are executed in parallel, the effects of an ongoing transaction must not be visible to other transactions

      如果交易是并行执行的，在一个正在进行交易的效果一定不会是可见的其他事务

  • Durability 耐用性

    • after a transaction finished successfully, its changes are persistent and will not be lost (e.g. on system failure)

      事务成功完成后，其更改是持久的，不会丢失（例如在系统故障时）

Money Transfer Example Revisited 汇款示例

  • Transaction to transfer money from account A to B

    1. start transaction
2. read(A)
3. A = A-700
4. write(A)
5. read(B)
6. B = B+700
7. write(B)
8. commit

  • Atomicity

    • if the transaction fails after step 4 but before step 8, the updates on A should not be reflected in the database (rollback)

      如果事务在步骤 4 之后但在步骤 8 之前失败，则 A 上的更新不应反映在数据库中（回滚）

  • Consistency

    • the sum of A and B should not be changed by the transaction

      A 和 B 的总和不应被交易改变

  • Isolation

    • if another transaction is going to access the partially updated database between step 4 and 7, it will see an inconsistent database (with a sum of A and B which is less than it should be)

      如果另一个事务将在步骤 4 和 7 之间访问部分更新的数据库，它将看到不一致的数据库（A 和 B 的总和小于应有的值）

  • Durability

    • once the money has been transferred from A to B (commit), the effect of the transaction must persist

      一旦资金从 A 转移到 B（提交），交易的效果必须持续存在

      • the only way to "undo" a committed transaction is to execute a compensating transaction

        “撤消”已提交事务的唯一方法是执行补偿事务

Transaction States 交易状态

  • Active 主动

    • initial state; transaction is in this state while executing

      初始状态；事务在执行时处于此状态

  • Partially committed

    • after the last statement has been executed

      执行完最后一条语句后

  • Committed

    • after successful completion

      成功完成后

  • Failed 失败

    • after discovery that a normal execution is no longer possible

      在发现不再可能正常执行后

      • logical error (e.g. bad input)

        逻辑错误（例如错误输入）

      • system error (e.g. deadlock)

        系统错误（例如死锁）

      • system crash

        或系统崩溃

  • Aborted 中止

    • after the rollback of a transaction

Transaction Management Components 组件

  • Transaction Manager 交易经理

    • ensures that we proceed from one consistent state to another consistent state (database consistency)

      确保我们从一种一致状态进入另一种一致状态（数据库一致性）

    • ensures that transactions will not violate integrity constraints

      确保事务不会违反完整性约束

  • Scheduler 调度器

    • provides a specific strategy for the execution of transactions and the corresponding concurrency control

      提供了具体的事务执行策略和相应的并发控制

    • avoids or resolves conflicts during concurrent data access

      避免或解决并发数据访问期间的冲突

  • Recovery Manager 恢复管理器

    • restore the database to the state it was in before a failure occurred (e.g. due to software bug or hardware problem) while executing one or multiple transactions

      在执行一个或多个事务时将数据库恢复到发生故障之前的状态（例如，由于软件错误或硬件问题）

Scheduler 调度器

Serial execution 串行执行

Serial execution of transactions 串行执行

  • each operation within a transaction can be executed atomically
  • any serial execution of a set of transactions T1,..., Tn by different users is regarded as a correct result

Parallel execution 并行执行

Parallel execution of transactions 并行执行

  • improves the throughput and resource utilisation as well as the average response time

    提高吞吐量和资源利用率以及平均响应时间

  • too much parallelism can lead to wrong results

    太多的并行性会导致错误的结果

    • e.g. dirty reads, lost updates, phantoms, ...

  • the scheduler has to choose the appropriate concurrency control scheme to avoid problems during parallel execution

    调度器必须选择合适的并发控制方案以避免并行执行时出现问题

A schedule S specifies the chronological order in which the operations of concurrent transactions are executed

调度 S 指定执行并发事务操作的时间顺序

  • a schedule for the transaction T1,..., Tn must contain all operations of these transactions

  • the schedule must preserve the order of the operations in each individual transaction

    调度必须保留每个单独事务中的操作顺序

Conflict Serialisability 冲突串行化

  • Two operations of transactions Ti and Tj form a conflict pair 冲突对 if at least one of them is a write operation

    交易的两个操作Ti和Tj形成冲突对，如果至少一个人是一个写操作

  • If a schedule S can be transformed into a schedule S' by a series of swaps of non-conflicting operations, then S and S' are conflict equivalent

    果调度 S 可以通过一系列非冲突操作的交换转换为调度 S' ，那么 S 和 S' 是冲突等价的

    • a conflict pair forces an order on the transactions

  • A schedule S is conflict serializable if it is conflict equivalent to a serial schedule

    如果调度 S与串行调度等价冲突，则调度 S 是冲突可序列化的

  • Transactions
    • T1: R1(x) W1(x) R1(z)
    • T2: R2(y) R2(x) W2(y)
    • T3: R3(y) R3(z) W3(z)
  • Schedule S
  • Conflict pairs
    • <W1(x),R2(x)>, <R3(x),W2(x)>, <W3(x),R1(x)>
  • Conflict equivalent schedule
  • Transactions
    • T1: R1(x) W1(x)
    • T2: W2(x)
  • Schedule S
  • Conflict pairs
    • <R1(x),W2(x)>, <W2(x),W1(x)>
  • This schedule is not conflict serializable!

Testing for Serialisability 可序列化测试

  • Check if there is a serial schedule with the same ordering of the conflict pairs as the given schedule S

    检查是否存在与给定调度 S 具有相同冲突对顺序的串行调度

  • Construct a precedence graph 优先图 G=(V,E) for S

    • the vertices V are represented by the transactions T1,..., Tn

    • there is an edge from Ti to Tj if there exists a conflict pair <x,y> in S with x ∈ Ti and y ∈ Tj and x is preceding y

    • a schedule S is serializable if its precedence graph is acyclic

      如果调度 S 的优先级图是非循环的，则它是可序列化的

  • Algorithm to find and equivalent serial schedule S'

    查找和等效串行调度 S' 的算法

    • construct the precedence graph for the schedule S

      构建调度 S 的优先级图

    • perform a topological sorting of the graph

      对图 进行拓扑排序

      • randomly choose a vertex with no incoming edges and remove the vertex and its outgoing edges from S (add its operations to S')

        随机选择一个没有传入边的顶点，并从 S 中删除顶点及其传出边（将其操作添加到 S'）

      • repeat the vertex removal until there are no more vertices or a cycle occurs

        重复顶点移除，直到没有更多的顶点或循环发生

Concurrency Control 并发控制

Different concurrency control schemes can be used to ensure that the isolation property is ensured when multiple transactions are executed in parallel

可以采用不同的并发控制方案，保证多个事务并行执行时的隔离性

The concurrency control schemes for implementing serialisation in an online system include

在线系统中实现序列化的并发控制方案包括

  • lock-based protocols

    基于锁的协议

  • validation-based protocols

    基于验证的协议

    • timestamp ordering 时间戳排序
    • optimistic concurrency control 乐观并发控制

  • graph-based protocols

    基于图的协议

Lock-based Protocols 基于锁的协议

One way to ensure serialisability is to require that data items can only be accessed in a mutually exclusive manner

确保可串行化的一种方法是要求数据项只能以互斥的方式访问

The DBMS has to offer a mechanism to lock a specific data object x for a given transaction Ti and mode m

DBMS 必须提供一种机制来为给定的事务Ti和模式m锁定特定的数据对象 x

  • lock(Ti , x, m)
  • unlock(Ti, x)

A transaction has to request a lock in the appropriate mode for a data object and can only proceed if the scheduler grants the lock

事务必须以适当的模式请求数据对象的锁，并且只有在调度程序授予锁的情况下才能继续进行

At any given time, there can never be two transactions with incompatible locks on the same data object

在任何时候，都不可能有两个事务在同一个数据对象上使用不兼容的锁

  • the second transaction has to wait until the object in unlocked

    第二个事务必须等到对象解锁

  • DBMS puts the waiting transactions into specific queues

    DBMS 将等待的事务放入特定队列

Types of locks

Current DBMSs implement two types (modes) of locks

当前的 DBMS 实现了两种类型（模式）的锁

  1. exclusive-mode lock (X) 排他模式锁
    • read and write access to the data object
  2. shared-mode lock (S) 共享模式锁
    • read-only access
    • at any time several shared-mode locks can be held simultaneously

lock is only granted if there is no other transaction that is already waiting for a lock on the same data object (to prevent starvation)

仅当没有其他事务正在等待同一数据对象上的锁定时才授予锁定（以*防止饥饿*）

  • Lock
    • exclusive lock Xi (x) and unlock Ui (x)

Two-Phase Locking 两相锁定 (2PL)

The two-phase locking protocol is based on two rules

两阶段锁定协议基于两个规则

  1. a data object has to be locked (exclusive or shared lock) before it can be accessed by a transaction (growing phase)

数据对象在被事务访问之前必须被锁定（排他或共享锁）（增长阶段）

  1. as soon as a transaction unlocks its first data object, it cannot acquire any further locks (shrinking phase)

一旦事务解锁了它的第一个数据对象，它就无法获得任何进一步的锁（收缩阶段）

The 2PL protocol guarantees serializable schedules

2PL 协议保证可序列化的调度

Problems of the 2PL protocol 问题

  • if we want to use the 2PL protocol, we have to know for each transaction Ti when no further locks will be necessary (lock point)

    如果我们想使用 2PL 协议，我们必须知道每个事务Ti 何时不需要进一步的锁（锁点）

    • not very realistic to have this kind of a priori knowledge

  • the 2PL protocol is not deadlock free

    2PL 协议不是无死锁的

  • potential problems in the case of an abort/rollback of an operation (cascading rollbacks)

    操作中止/回滚的潜在问题（级联回滚）

2PL Cascading Rollback Problem 级联回滚

The 2PL protocol is not suited to handle transactions that are aborted since cascading rollbacks may occur

2PL 协议不适合处理中止的事务，因为可能会发生级联回滚

  • Cascading rollback 级联回滚

    • transaction Ti ends with an abort/rollback operation A

      事务Ti以中止/回滚操作 A 结束

    • this might trigger a previously committed transaction Tj to be aborted too!

      这可能会触发先前提交的事务Tj也被中止！

  • Abort example 中止示例
    • An abort A1 of transaction T1 can result in an abort of transaction T2 since it got access to "shared" data objects after T1 reached its lock point

2PL Deadlock Problem 死锁问题

Two transactions Ti and Tj might wait in a cycle for a lock held by the other transaction

两个事务Ti和Tj可能在一个周期内等待另一个事务持有的锁

The scheduler has to periodically check whether such cycles (deadlocks) exist

调度器必须定期检查是否存在这样的循环（死锁）

  • use a directed wait-for graph to model the dependencies between transactions

    使用有向等待图对事务之间的依赖关系建模

  • if a deadlock occurs (cycle in the wait-for graph), the scheduler has to reset one of the participating transactions

    如果发生死锁，则调度程序必须重置其中一个参与事务

Example

Strict Two-Phase Locking 严格两相锁定 (S2PL)

The strict two-phase locking protocol is based on two rules

严格的两阶段锁定协议基于两个规则

  1. a data object has to be locked (exclusive or shared lock) before it can be accessed by a transaction (growing phase)

数据对象在被事务访问之前必须被锁定（排他或共享锁）（增长阶段）

  1. a transaction keeps all locks until the end of the transaction and releases them all at once (commit/abort phase)

事务保持所有锁直到事务结束并立即释放它们（提交/中止阶段）

  • The S2PL protocol guarantees serializable schedules

    S2PL 协议保证可序列化的调度

    • S2PL avoids cascading aborts but it is still not deadlock free

      S2PL避免了级联中止，但它仍然不是无死锁

  • S2PL is implemented in every major database system

    S2PL在每个主要的数据库系统中都实现了

  • S2PL can be implemented in a deadlock-free manner (deadlock prevention)

    S2PL可以以无死锁的方式实现（死锁预防）

    • transaction has to acquire all necessary locks (proclaiming of locks) before the first operation is executed

      在执行第一个操作之前，事务必须获得所有必要的锁（声明锁）

    • reduces potential concurrency for long transactions

      减少长事务的潜在并发

Locking Granularity 锁定粒度

Locking protocols such as 2PL or S2PL can be applied at various granularity levels

2PL 或 S2PL 等锁定协议可应用于各种粒度级别

  • pages/blocks 页/块
    • commonly used
  • relations 关系

    • very coarse and restrictive

      非常粗糙和限制

  • tuples

There is a trade-off between concurrency and overhead

并发性和开销之间存在权衡

  • Small granularity 小粒度

    • higher level of potential concurrency but larger locking overhead

      更高级别的潜在并发但更大的锁定开销

  • Large granularity 大粒度

    • less potential for concurrency but smaller locking overhead

      并发的可能性较小，但锁定开销较小

Validation-based Protocols 基于验证的协议

Rather than to prevent conflicts from the beginning, it is sufficient to detect them and resolve them

与其从一开始就预防冲突，不如发现并解决冲突

  • abort transaction in the case that a conflict is detected

    在检测到冲突的情况下中止事务

  • works efficiently if the probability for conflicts is very low

    如果冲突的可能性非常低，则工作效率高

Example scheduling techniques - timestamp ordering 时间戳排序 - optimistic concurrency control 乐观并发控制

Timestamp Ordering 时间戳排序

Each transaction Ti gets a unique timestamp TS(TSi) assigned

每笔交易都会获得一个唯一的时间戳

  • e.g. based on system clock or a logical counter

    例如基于系统时钟或逻辑计数器

Timestamp ordering rule 时间戳排序规则

  • if operation oi,m(x) is in conflict with operation oj,n(x) and oi,m(x) is part of transaction Ti whereas oj,n(x) is part of Tj , then they have to be ordered oi,m(x) < oj,n(x) if TS(Ti) < TS(Tj)

  • the timestamp order defines the serialisation order

    所述时间戳顺序定义的串行次序

For each access of a data object the scheduler has to check whether a later transaction (larger timestamp) has already accessed the object

对于数据对象的每次访问，调度程序必须检查后面的事务（更大的时间戳）是否已经访问了该对象

  • W-TS(x): largest timestamp of transaction writing x
  • R-TS(x): largest timestamp of transaction reading x
Timestamp Ordering Scheduler 时间戳排序调度器
  • Ti wants to read object x
    • TS(Ti) < W-TS(x)
      • Ti wants to read old data → reset Ti
    • TS(Ti) ≥ W-TS(x)
      • permit read operation and update R-TS(x)
  • Ti wants to write object x
    • TS(Ti) < R-TS(x)
      • there is a newer transaction that already read x → reset Ti
    • TS(Ti) < W-TS(x)
      • Ti wants to write an obsolete value → reset Ti
    • TS(Ti) ≥ R-TS(x) and TS(Ti) ≥ W-TS(x)
      • permit write operation and update W-TS(x)

Optimistic Concurrency Control 乐观并发控制 (OCC)

  • Assumes that there will not be many conflicts

    假设不会有很多冲突

  • Transactions are executed with the explicit risk of abortion

    交易的执行存在明显的流产风险

    • in snapshot isolation each transaction has a private workspace

      在快照隔离中，每个事务都有一个私有工作区

Three phases of a transaction Ti

  • reading and execution phase 读取和执行阶段

    • Ti reads objects of the database (read set of Ti) and writes private versions (write set of Ti)

  • validation phase 验证阶段

    • before writing the private versions to the database a conflict analysis is performed

      在将私有版本写入数据库之前执行冲突分析

  • writing phase 写作阶段

    • if the validation was positive the private versions are written to the disk

      如果验证是肯定的，私有版本将写入磁盘

Recovery 恢复

The recovery manager has to ensure that the atomicity and durability properties are preserved in the case of a system failure

在恢复管理器必须确保原子性和耐久性特性在系统发生故障的情况下保存

Different types of system failures - transaction failure 交易失败

  • logical error 逻辑错误

    • internal transaction problems (e.g. bad data input or data not found)

      内部事务问题（例如错误的数据输入或未找到数据）

  • system error 系统错误

    • system in an undesirable state (e.g. deadlock)

      系统处于不良状态（例如死锁）

  • system crash 系统崩溃

    • software bug or hardware malfunction not affecting the non-volatile storage

      不影响非易失性存储的软件错误或硬件故障

  • disk failure 磁盘故障

    • content loss on non-volatile storage (e.g. data transfer error or head crash)

      非易失性存储上的内容丢失（例如数据传输错误或磁头崩溃）

Log-based Recovery 基于日志的恢复

After a system failure we should be able to return to a state where ongoing transactions have either been successfully completed or had no effect on the data at all

系统故障后，我们应该能够返回到正在进行的事务已成功完成或对数据根本没有影响的状态

To support undo and redo operations, we must write logging information to a stable storage before modifying the database

为了支持撤销和重做操作，我们必须在修改数据库之前将日志信息写入一个稳定的存储中

  • atomicity 原子性

    • undo based on logging information

      根据日志信息撤消

  • persistency 持久性

    • redo based on logging information

      基于日志信息重做

  • redo and undo operations must be idempotent

    重做和撤消操作必须是幂等的

    • executing them multiple times has the same effect as executing them once

      基于日志信息重做

A log file consists of a sequence of log records which can be of the following types

一个日志文件包含一个的日志记录顺序可以是以下类型

  • start of transaction Ti
    • <Ti start>
  • transaction Ti updates the value V1 of data item Xj to value V2
    • <Ti , Xj , V1, V2>
  • commit of transaction Ti
    • < Ti commit>
  • abort of transaction Ti
    • < Ti abort>

We can either perform an immediate or a deferred database modification

我们可以执行立即或延迟的数据库修改

Immediate Modification Technique 立即修改技术

  • Allows uncommitted database modifications while the transaction is still in its active state

    当事务仍处于活动状态时允许未提交的数据库修改

  • Before a transaction Ti starts, we write the corresponding start record to the log

    在事务Ti 开始之前，我们将相应的开始记录写入日志

  • Each write operation is preceded by the corresponding update record in the log

    每次写操作前面都有日志中对应的更新记录

  • When the transaction Ti partially commits, we write the corresponding commit record to the log

    当事务Ti 部分提交时，我们将相应的提交记录写入日志

  • To support concurrent transactions we further have to ensure that there are no conflicting update operations (e.g. by using S2PL)

    为了支持并发事务，我们还必须确保没有冲突的更新操作（例如，通过使用 S2PL）

Logging Example 日志示例

The two transactions might result in the following log file (annotated with the database state)

Restart Recovery 重启恢复

After a crash the system scans the log file and constructs two lists of transactions

崩溃后系统扫描日志文件并构建两个事务列表

  • redo-list 重做列表

    • contains each transaction Ti for which a log record exists

      包含存在日志记录的每个事务

  • undo-list 重做列表

    • contains each transaction Ti for which no log record exists

      包含不存在日志记录的每个事务

The system then performs the following two steps

系统然后执行以下两个步骤

  1. scan the log file in reverse order and for each log record of a transaction Ti in the undo-list perform an undo operation

    以相反的顺序扫描日志文件，并对撤消列表中事务Ti的每个日志记录执行撤消操作

  2. scan the log file in forward order and for each log record of a transaction Ti in the redo-list perform a redo operation

    以正向顺序扫描日志文件，并为重做列表中事务Ti的每个日志记录执行重做操作

To avoid a scan of the entire log file, special checkpoint operations can be performed periodically

为了避免扫描整个日志文件，可以定期执行特殊的检查点操作
