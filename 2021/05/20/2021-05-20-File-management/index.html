File management

Video + Slide

在本章中，将讨论存储设备和文件的组织。提供非常大的逻辑地址空间logical address space（数量级为 264 字节byte）并将所有设备视为main memory的虚拟扩展virtual extension，这是完全可能的，并且在某些系统中已完成此操作。

在这种情况下，外围存储器peripheral memory不需要特殊的管理系统，因为所有存储器的管理将由中央存储器管理系统central memory management system提供。

然而，这种方法并不普遍，绝大多数操作系统将外围存储器的（小）部分作为主存储器的虚拟扩展单独管理，其余部分作为用户和应用程序按名称访问的一组文件进行管理。

本章将介绍广义的文件管理。

Physical disk organization

正如在本课程第一部分的第 1 章中已经提到的，用于存储信息的磁盘magnetic disk由几个围绕公共轴common axis高速旋转的磁性表面magnetic surface组成。信息是使用磁头magnetic head写入和读取的，磁头magnetic head需要紧邻磁性表面magnetic surface（实际上，在空气动力aerodynamic的作用下，它们漂浮在表面上方几微米micron的高度altitude）。

每个magnetic surface都由一个magnetic head“飞行flown”。所有magnetic head都安装在可移动的梳齿comb上，用于将magnetic head定位在多个预定义predefined的圆形circular磁路magnetic path上。

与轴axis等距的不同magnetic surface上的一组磁道track称为“圆柱体cylinder”。同时访问同一柱面cylinder的所有track，无需移动磁头magnetic head。

信息作为几千位bit的独立记录写入磁盘。这样的记录称为“扇区sector”。如果您知道区域area、柱面cylinder和它所在的区域，则可以在磁盘中定位特定记录。通常，这种三维寻址被通过对所有记录进行编号而获得的一维地址所代替，以便尽可能快地以数字顺序访问磁盘的物理特性。

Clusters vs. Sectors

人们可能想知道磁盘的最佳扇区大小是多少what is the optimal sector size of a disk。

小区域small area的主要优点是减少浪费的空间（通常称为“松弛slack”），因为需要在每个文件中分配整数个扇区sector。在大文件的情况下，我们统计每个文件丢失半个sector。在小文件的情况下，可能会丢失相当多的地方。因此，在低容量驱动器low-capacity drive的情况下，最好选择小扇区small sector size大小。

大扇区大小large sector size的主要优点是由于传输块大小size of block的增加而增加了数据吞吐量data throughput。另一个优点在于扇区sector的编号numbering：扇区sector越大，对于给定的总容量total capacity，sector数越少，从而减少扇区号sector number所需的bit数。对于较大的磁盘，首选大扇区。

  • What is the optimal size of a disk sector ?
  • Small sectors : good for small disks
    • Reduce slack
  • Large sectors : good or large disks
    • Increase disk throughput
    • Reduce number of bits in sector number
  • It is desirable to have a single sector size for all disks
  • Solution : define clusters as groups of sectors with size optimized for each specific disk.
  • Transfer entire clusters instead of sectors
  • Each cluster on disk has a Physical Cluster Number ( PCN )

然而，由于磁盘控制器disk controller及其驱动程序driver的复杂性，无论磁盘是什么，都非常希望对扇区采用统一的大小。

这就是为什么在包括微软在内的一些系统中引入了“集群cluster”一词，它是一组连续consecutive的物理扇区sector，软件将其视为一个扇区sector，从软件传输整个cluster的意义上说，并管理cluster编号而不是物理扇区sector数。

尽管术语“集群cluster”实际上是 Microsoft 系统独有的，但它将在接下来的解释中用作可识别的最小数据块smallest identifiable data block的名称，并可由文件管理软件传输。

Formats of 3.5" diskettes for PC's

旧 PC 软盘floppy disk提供了一个有趣的说明，说明了可能决定cluster大小的相互竞争的考虑因素。

3.5"软盘有两种格式：格式"Double Density"和格式"High Density"。下表显示了cluster采用的物理特性和大小。

Double Density High Density
Tracks / cylinder 2 2
Cylinders 80 80
Sectors / track 9 18
Bytes / track 512 512
Total capacity (in bytes) 737280 1474560
Sectors / Cluster 2 1

双密度double density软盘floppy disk采用的是2 sector的cluster，而高密度high-density软盘disk则是一个sector的cluster。

这些奇怪选择的原因是历史性的：在很少有个人计算机配备硬盘驱动器的时候，使用了双密度软盘。所有软件和所有数据都存储在软盘上。因此，磁盘的transfer speed对系统性能至关重要，采用相当大的cluster是合乎逻辑的。

推出高密度软盘时，所有计算机都配备了硬盘驱动器hard drive。软盘仅用于备份或数据交换。因此，对于这些新磁盘，采用cluster大小以最大限度地减少“slack”中浪费的空间是合乎逻辑的。

Logical to physical mapping

在文件管理领域要解决的第一个主题是文件的逻辑组织和物理组织之间应该存在的关系。

由于物理上完整的文件存储为一组固定大小的记录，因此自然也将用户已知的logical file视为大小与physical record size相同的块组sets of blocks。 此类block在本章的其余部分称为“集群clusters”。

每个cluster都有一个物理簇号 (PCN)，由它在磁盘上的物理位置确定，还有一个逻辑簇号 (LCN)，由它在所属文件中的逻辑位置确定。

下图介绍了主题，是建立文件簇的逻辑编号logical number和物理编号physical number之间的对应关系。

根据所讨论的操作系统，以各种方式实现对应关系。

DOS & Windows: File Access Table (FAT16)

Stream and free file

在微软的 DOS 中，每个文件，也称为“流stream”，是一个cluster序列。每个cluster包含 512 byte数据的倍数，并且每个cluster都与一个指针pointer相关联，该pointer给出了文件下一个cluster的物理编号。该指针可能是indicate the end of a file的particular value。

我们看到 DOS 文件基本上是顺序sequential文件，因为对任何文件的探索都必须从第一个cluster开始（或至少从与第一个cluster关联的指针开始）。

整个光盘中存在一个特殊文件（“free”），其中包含所有未使用的cluster。当一个文件被删除时，它实际上只是简单地插入到free file之前，当一个新文件被创建时，它的cluster从free file中取出。

从这一观察我们可以推断出，在文件销毁期间，数据通常不会立即丢失，只有当属于被销毁文件的一个cluster被回收到另一个文件时才会丢失。

如果发生unwanted的文件删除，recovery首先要避免创建新文件并使用允许更改与不同cluster关联的pointer的计算机工具。

File Access Table

在 DOS 系统中，与磁盘的不同cluster相关联的指针不是cluster本身的一部分。它们被分组在一个特定的表中，包括两个副本存储在每个磁盘上。 该表是“文件访问表File Access Table”，通常缩写为“FAT”。

FAT 由两个独立副本independent copy保留，因为在擦除 FAT 后，存储在磁盘上的所有数据实际上都无法使用。 不幸的是，丢失 FAT 的最常见原因是病毒的恶意干预，而病毒编写者知道一张光盘上总是有两个 FAT 副本。

Minimum cluster size

FAT 中包含的pointer长度为 16 位，这限制了磁盘 216 (65536) 上的cluster数。 如图所示，此限制是使用容量可达到几十GB 的现代磁盘的严重情况。

Disk size Sectors / Cluster Cluster size
128 MB 4 2 kB
256 MB 8 4 kB
512 MB 16 8 kB
1 GB 32 16 kB
2 GB 64 32 kB
4 GB 128 64 kB
8 GB 256 128 kB

我们看到，对于 8 GB 磁盘cluster大小至少为 128 KB。 这也意味着任何文件，即使它只包含几个字节，也必须具有 128 KB 的最小大小。

这种行为解释了为什么当磁盘容量超过 1 GB 时，DOS 用户倾向于将磁盘分区partition为两个、三个或四个partition，由 DOS 管理，就好像它们是单独的磁盘一样。

FAT32

FAT16 became inadequate as soon as the size of disks started to exceed 1 GB.

  • Cluster number = 32 bits instead of 16 bits

  • Reasonable cluster sizes again possible

  • Size of fat becomes an issue

  • Each storage device has its own fat

  • FAT16 orFAT32 depends on size of device

尽管新的磁盘管理模式称为 FAT32，但在 Windows 系统中通过放弃事实上的 FAT 解决了此问题。 与此处描述的 FAT 中使用的 16 位指针不同，原始 FAT 现在称为 FAT16。

Unix - Linux: inode

在 UNIX 和 LINUX 中，物理集群和逻辑集群之间的对应关系不像 DOS 那样针对整个驱动器建立，而是通过每个文件的单独表建立。该表名为“inode”，包含 13 个物理簇号。

前 10 个是前 10 个逻辑文件簇的物理编号。第11个是物理簇号，包含了后面256个逻辑簇的物理号。第 12 位是一个簇的物理编号，它包含 256 个簇的物理编号，这些物理编号依次包含 256 × 256 个以下逻辑簇的物理编号。最后，第十三个，根据类似的机制，允许访问文件的最后一个 256 × 256 × 256 簇。

很明显，只有访问有效填充的文件簇所需的树结构部分才会占用磁盘空间。与在 DOS 中发生的情况不同，这允许创建非常小气stingy的非常大 large的文件，而不会不必要地占用磁盘空间。

inodes结构以非常明显的方式促进对短文件和长文件的开头的访问。该选项在统计上非常适合办公自动化和软件工程的应用。

Windows NTFS: Master File Table

为了消除FAT16的限制，微软设计了一个完全不同的文件系统（New Technology File System）。 在该系统中，簇号从 16 位传递到 32 位。

为了减少 FAT 等价物equivalent的大小，考虑了经常将连续的物理地址分配给逻辑上连续的簇的事实。

在 NTFS 中，每个文件都有一个表，其中包含逻辑簇编号、相应物理簇的编号以及此关联适用的连续簇的编号。

可以从这种组织中得出结论，MS/DOS 文件的基本顺序性质得到了保护，但更实用和更紧凑的表格已经取代了旧的 FAT16。

本章稍后将阐明这些新表的位置。

Disk directories

Directory Tree

除了物理和逻辑簇数之间的对应关系，文件管理器还管理用户操作的文件命名。为此，文件管理器使用至少包含每个文件名称的目录directory和一个pointer来查找显示这些文件的逻辑簇和物理簇之间对应关系的表。

绝大多数当前的管理系统使用树目录 tree-directory，其中目录与其他文件相似。

上图显示了一个目录树tree-directory（MS / Windows 独有）：顶部是初始目录，称为“root”并注明 \，其中包含对 8 个文件的引用。其中五个以绿色显示并设计为方角的文件包含data or program，而其他三个以橙色表示并用圆角绘制的文件本身就是目录。

在这样的结构中，文件的专有名称前面是所有目录的列表，这些目录必须从初始目录经过才能到达专有名称。这种树结构允许通过按层次结构中的主题对文件进行分组来管理大量文件。

Directory Entry

例如，下图显示了DOS 目录的结构。

对于每个文件，有一个 8 字节的名称和三个字节的名称扩展名。名称的扩展名在 DOS 中传统上用于指示文件内容的性质。因此，.exe 扩展名用于包含可执行程序的目标代码的文件，而 .ppt 扩展名则以使用 PowerPoint 程序创建的演示文稿为特色。

创建文件的日期和时间（或更准确地说，是内容的最后修改日期）包含在目录中，以帮助识别同一文件的连续版本。

显然，找到了指向第一个物理簇的指针，并指示以字节表示的文件总长度。磁盘上的文件大小是超过此长度的簇大小的第一个倍数。

最后是一个字节，每一位都有一个特殊的含义，称为文件的“属性attribute”。属性字节的第 4 位将普通文件与目录区分开来。其他属性允许保护某些文件免受意外或恶意更改或防止在目录中显示其名称

最后，与某些备份程序结合使用的存档位允许在备份期间仅复制自上次备份以来已更改的文件。

Unix directories

UNIX 和 LINUX 系统也使用树目录（实际上，这种目录流行的是 UNIX），但是 UNIX 树的语义稍微复杂一些，因为同一个文件可以在多个目录中引用，因此有多个不同的名字。

例如，当一个文件用于多个不同的项目时，这种可能性可能会很有趣，并且当共享文件的内容发生变化时，我们不想使用副本来避免一致性问题。

此类共享文件的管理引发了一系列有趣且复杂的问题，其中可以举出一些特别复杂的示例：

  • 我们如何避免无意中创建的引用循环会导致目录读取程序（例如 ls）陷入无限循环？

  • 当文件的用户决定销毁它而其他人仍然需要它时，我们应该怎么做？

  • 如果要等待一个所有用户都销毁的文件占用的空间再回收，我们如何验证是否如此？

MS/NTFS Master File Table

Microsoft 的 NTFS 文件管理器为其目录采用了原始格式：每张光盘都包含一个名为“主文件表Master File Table”的大表。

在 MFT 中，每个文件占用 1. 2 或 4 KB 的空间。在这个空间中有几个大小和数量可变的字段，用于描述文件，可能是为了确保它们的安全并允许找到它的内容。

因此，每个文件可以有多个名称：传统 DOS 格式中的名称为 8 + 3 个字符character，Windows 标准的名称长度不受限制，甚至可能是其他管理系统特有的名称，这些名称可以在 Windows NT 环境中模拟。

在其他领域，我们可以找到已应用于文件内容的加密和/或压缩技术的描述或其属性的非常详细的列表。

最后，在最后一个字段中，有与文件实际内容相关的信息。此信息的性质和组织可能因文件而异。

MS/NTFS Data Files

如果相关文件是小数据文件，则直接存储在该文件对应的MFT块中。 通过这种方式，我们在查看小文件时避免了额外的磁盘访问，因为在任何 Windows 系统中都有数千个文件。

另一方面，如果它是一个大文件，在 MFT 中没有足够的空间，那么逻辑和物理集群对应表将被放置在 MFT 中，如前所述，允许从 MFT 中包含的数据中有效地查找， 整个文件的内容。

MS/NTFS directories

目录使用有点类似的策略。 对于小目录，MFT 中所有列出文件的索引都包含在 MFT 块中，即目录directory。

另一方面，如果它是一个大目录，则它被组织为称为“BTREE”的树结构，其中只有根和包含结构其余部分的物理簇的编号包含在 MFT 中。

使用 BTREE 表示大目录可以显着缩短在目录中查找文件所需的时间（在传统文件中，此时间与目录中的文件数成正比，而在大型 NTFS 目录中，此时间与此数量的对数成正比）

System calls

文件管理是计算机操作系统的任务之一。所以操作系统必须包含一系列可以被应用程序调用来执行文件操作的函数。

操作系统通常提供的功能是：

  • 使用用户提供的名称创建和销毁文件。create file, delete file.
  • 打开和关闭现有文件。Open, close.
    • 打开文件包括在目录中搜索名称并创建将由驱动程序用来传输数据的内存缓冲区。
    • 关闭文件基本上是刷新任何缓冲区并将它们写入磁盘上的文件中。
    • 当程序不小心关闭时，可能是文件处于打开状态，磁盘上的物理和逻辑簇之间的对应表没有更新，因此文件处于不正确的状态。这就是为什么在每次计算机意外关闭后，操作系统都应该检查整个磁盘以恢复潜在的无法访问的集群。
  • 应用程序和相关磁盘缓冲区之间的数据传输，磁盘缓冲区和磁盘之间的传输通过直接访问控制器内存独立进行。
  • 将顺序文件定位到文件中指定的位置 reposition.
  • 读取或更改文件的属性。get file attributes, set file attributes
