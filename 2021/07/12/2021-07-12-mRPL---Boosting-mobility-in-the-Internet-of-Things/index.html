mRPL: Boosting mobility in the Internet of Things

VUB 4020724ENR Advanced IT Networks

Hossein Fotouhi, Daniel Moreira, Mário Alves, mRPL: Boosting mobility in the Internet of Things, Ad Hoc Networks., 26 (2015), pp. 17-35

Fundamental building block for the IoT

然而，随着最先进技术的进步，学术和商业努力发明了基于互联网架构的新网络抽象。

  • The Internet Engineering Task Force (IETF) 设计了一些protocols和适配层adaptation layers，允许 IPv6 在 IEEE 802.15.4 链路层link layer上运行。
  • IPv6 over Low-power Wireless Personal Area Networks (6LoWPAN) 工作组设计了 IPv6 over IEEE 802.15.4 的报头压缩header compression和分段fragmentation。
  • IETF Routing Over Low-power and Lossy networks (ROLL) 工作组设计了一种路由协议，简称 RPL，它是 6LoWPAN 的事实上的标准路由协议。

因此，这些基于 IP 的标准协议是物联网的基本构建块。

Routing scheme 路由方案

6LoWPAN 作为适配层adaptation layer，设计了mesh-under和route-over两种路由方案，分别支持链路层和网络层路由。

  • Mesh-under design

    • The adaptation layer performs the mesh routing and forwards packets to the destination via multiple radio hops.

      适配层执行网格路由并通过多个无线电跳将数据包转发到目的地

    • 适用于单跳网络 single-hop networks

      • 有节点都在彼此的传输范围内

  • Route-over

    • 路由决策在网络层做出
      • 节点充当 IP 路由器
    • 每个链路层link layer跃点hop都是一个 IP hop，IP 路由在这些link之间转发数据包。
    • 支持 multi-hop mesh routing communication，适合大规模部署。

Mobility support 移动性支持

许多应用程序需要及时性timeliness和可靠性reliability保证从源到目的地传输关键消息，但在低功耗和移动网络中提供服务质量Quality of Service (QoS) 是非常具有挑战性的。

本文作者考虑了一种收集患者生命体征的无线临床监测应用程序。患者是移动节点mobile nodes，可产生流量traffic并自由移动，同时保持与固定节点基础设施的连接。该系统模型中的所有节点都是具有低功耗 CC2420 无线电的简单传感器节点。

Mobility detection 移动性检测

移动性被认为是 RPL 不一致的主要来源之一。

通常，有两种主要方法可以帮助检测移动性；

  1. 由 Trickle 算法控制的 ICMPv6数据包传输

    • 首先，由 Trickle 算法调度的控制数据包control packets的周期性传输periodic transmission可以检测拓扑topological变化。

    • 在此过程中，RPL 会恢复导致高开销的快速全局路由更新。

      mRPL 性能与 Trickle 设置无关。

      分析表明，较高的数据流量会导致较低的数据包传输率。

      Trickle 计时器的较小值和较高的数据生成速率会增加网络流量，从而增加数据包冲突的机会。数据包丢失在较大的涓流计时器中更为显着

      Trickle 计时器的较小值会在 RPL 中施加较高的控制数据包开销。开销按照ICMPv6报文占报文总数的百分比（ICMPv6报文 + 数据报文）计算

      较低的数据传输速率会导致控制消息相对于数据包交换总数的百分比较高。

  2. 由 ND 协议控制的 ICMPv6 数据包传输。

    • 其次，邻居发现（ND – 在 RFC 4861 中定义）机制定期评估邻居可达性。
    • 在每次激活时，ND 协议都会用路由器广告淹没整个网络，这也导致了很高的开销。
      • 较短的激活间隔（减少开销）导致对网络/拓扑变化的低响应性。
    • 但是，在 6LoWPAN 修改后的 ND 机制中，路由器广告数据包是在收到路由器请求消息时发送的

Trickle algorithm

低功耗网络中的传统收集协议通常以fixed time interval广播控制消息。

  • 小间隔需要更多的带宽和能量。
  • 小间隔使用较少的带宽和能量，但由于无法应对网络动态，可能会出现拓扑问题。

Trickle 算法（在 RFC 6206 中定义）的基本思想是在路由信息发生变化时传播信标。RPL 通过使用基于涓流Trickle-based的计时器降低了传播路由状态的成本。

Trickle 是一种自适应adaptive信标beaconing策略，旨在快速恢复和低开销。

虽然 DIS 数据包会定期从路由器发送，直到选择第一个父节点，但使用 Trickle 计时器来调度 DIO 的传输。此计时器允许 DIO 间隔在网络条件稳定时呈指数增加，并在检测到网络条件发生明显变化时迅速减小到最小值。

周期性Trickle timer t 受间隔限制[Imin; Imax]

  • Imin

    • 由 base-2 值定义的最小间隔（以毫秒为单位）

      e.g. 212 = 4096 ms

  • Imax

    • Imax = Imin × 2 Idoubling用于限制Imin可以加倍的次数

      假设Idoubling = 4，Imax = 4096 × 24 = 65536 ms

Trickle 算法能够在短时间内全局维护拓扑更新。

  • 检测到 DIO 消息中的不一致（例如由节点移动性强加）的节点，将 t 设置为 Imin 并更新树。
  • 如果 DODAG 保持一致，则每次 DIO 传输发生时 t 都会加倍，直到达到 Imax，保持该值不变。 当网络稳定时，Trickle 定时器逐渐收敛到它的最大间隔。

在移动时，这个大的间隔会导致非常低的网络响应。 在检测到网络中的任何不一致后，网络中所有节点的 DIO 周期呈指数下降，从而影响网络开销。

IPv6 neighbour discovery approach

RPL 可以使用IPv6 neighbour discovery approach来检测环境变化。 低功率链路利用优化版本的 ND，该版本由 IETF 开发，作为 6LoWPAN 邻居发现的改编版本。

ND 协议允许节点检测邻居不可达性并发现新邻居。

该协议由四个 ICMPv6 控制消息支持：

  1. Neighbour Solicitation (NS)
    • 确定邻居的链路层地址并验证邻居是否仍然可达
  2. Neighbour Advertisement (NA)
    • 回复 NS 消息，并定期发送以通知链接更改，
  3. Router Solicitation (RS)
    • 从主机节点（我们系统模型中的移动节点）向其路由器请求信息
  4. Router Advertisement (RA)
    • 路由器定期发送并作为对 RS 消息的响应，该消息通过链路信息和 Internet 参数来通告其（路由器的）存在

Smart-HOP

Why smart-HOP?

切换Hand-off在蜂窝Cellular和无线局域网wireless local area networks中得到了广泛的研究。然而，它在低功耗网络RPL中并没有受到同等程度的关注。

  • 蜂窝网络执行集中切换决策，通常由强大的基站协调。与蜂窝网络相反，
  • WiFi 网络具有分布式架构，当服务质量下降时会触发切换。

In low-power networks, a centralized approach is not feasible as the access points are assumed to have scarce resources.

在低功耗网络中，集中式方法是不可行的，因为假设接入点资源稀缺。

smart-HOP, considers the main features of low-power networks, the link unreliability and the existence of a single low-power radio per node. It manages hand-offs in a distributed way and leads to very short disconnection times.

smart-HOP 考虑了低功率网络的主要特征、链路不可靠性以及每个节点存在单个低功率无线电。它以分布式方式管理切换并导致非常短的断开时间。

Why integrating smart-HOP in RPL?

  • RPL 的主动特性proactive feature，可以生成和维护稳定的路由表。

    • 所有节点之间的控制消息的周期性广播periodic broadcast维护它们之间的路径和链路状态。

    • In reactive routing protocols; such as AODV and DSR, routes are established upon request, so they do not respond quickly to environmental changes due to mobility or link degradation.

      在反应式路由协议中；例如 AODV 和 DSR，路由是根据请求建立的，因此它们不会对由于移动性或链路退化引起的环境变化做出快速响应。

    • RPL 以最小的开销在后台维护路由，对于移动性有限且对基础设施有要求的应用，RPL非常适合。

  • unlike other proactive routing protocols (e.g. OSPF), RPL exchanges local information among neighbours to repair routing inconsistencies, instead of globally advertising control messages,

    与其他主动路由协议不同，RPL 在邻居之间交换本地信息以修复路由不一致，而不是全局广告控制消息。

  • RPL 基于树的结构，适合数据收集 WSN 应用程序。

  • RPL 中基于 IPv6 的寻址自然地执行与其他 Internet 设备的互操作性。

Phase 阶段

smart-HOP 算法有两个主要阶段：

  1. Data Transmission Phase 数据传输阶段
  2. Discovery Phase 发现阶段

为了清楚起见，让我们假设一个节点处于数据传输阶段。在此阶段，假设移动节点mobile node (MN) 与接入点access point (AP) 有可靠链路，接入点 (AP) 定义为Serving AP。

  • Mobile node通过接收来自serving AP的回复数据包来监控链路质量。
  • 在给定窗口中接收到 n 个数据包后，serving AP会回复 n 个数据包的平均 RSSI (ARSSI) 或 SNR。
  • 如果没有收到数据包，serving AP不采取任何行动。
  • 这可能会导致断开连接，这可以通过使用超时机制来解决。

It is important to notice that smart-HOP filters out asymmetric links implicitly by using reply packets at the Data Transmission and Discovery Phases.

值得注意的是，smart-HOP 通过在数据传输和发现阶段使用回复数据包隐式过滤掉非对称链路。

如果相邻 AP 没有活动链路，则该 AP 根本不参与该过程。

Parameters 主要参数

  • Window size (ws)
  • Hysteresis margin 滞后余量 (HM)
  • Stability monitoring (m)

Window size (ws)

ws是在特定时间间隔内计算 ARSSI 所需的数据包数量，如图所示。

一个小的ws提供了有关link的详细信息，但会增加对回复数据包的处理，从而导致更高的能耗energy consumption和更低的交付率delivery rate。

随着 MN 选择执行一些不必要的切换hand-offs，数据包传递会减少。

  • 切换hand-offs是通过检测到信号强度降低导致的低质量链路来触发的。

另一方面，较大的ws提供有关链路的粗粒度信息coarse grained information并降低系统的响应能力，这不适用于具有动态链路变化的移动网络。

Hysteresis margin 滞后余量 (HM)

定义: 开始切换 (Tl) 的 ARSSI 阈值和停止切换的 ARSSI 阈值之间的差值。Th = Tl + HM

在 WSN 中，阈值和滞后余量hysteresis margins的选择取决于过渡区域的特性和无线链路的可变性。

  • 应根据过渡区域transitional region的边界选择阈值

    • 如果 Tl 阈值太高，节点可能会执行不必要的切换（因为选择性太大）

      过渡区域的大小通常相当大，因此网络中的大量链接（高于 50%）是不可靠的

      • 因此，无线节点很可能大部分时间都在过渡区域

    • 如果阈值太低，节点可能会使用不可靠的链接

  • 滞后余量在应对低功率无线链路的可变性方面起着核心作用

    • 如果滞后余量太窄
      • 移动节点可能最终会在两个 AP 之间执行不必要且频繁的切换（乒乓效应ping-pong effect）
    • 如果滞后余量太大
      • 切换可能需要太长时间，这最终会增加网络不可达时间，从而降低交付率。

Stability monitoring (m)

由于无线链路的高度可变性，移动节点可能会检测到瞬间高于Th的 AP，但在切换到该 AP 后不久 ARSSI 可能会下降。为了避免这种情况，重要的是评估候选 AP 的稳定性。

在检测到 ARSSI 高于 Th 的 AP 后，MN 继续 m 个进一步的发现阶段以检查该 AP 的稳定性。

可以很容易地推断出，稳定性监控stability monitoring和滞后裕度参数hysteresis margin parameters是紧密耦合的。

  • 较宽的滞后余量需要较低的 m，反之亦然。
  • 适当调整滞后余量将导致 m = 1，从而使开销最小。

mRPL算法

该算法的时间线如图所示。

在这种方法中，MN 在传输预定数量的数据包（窗口尺寸）。

DIO 回复消息（保持平均 RSSI 级别）隐式过滤掉非对称链接。

  • 在检测到质量良好的链路时（根据 DIO 回复消息中的平均 RSSI 级别），MN 继续数据传输阶段。
  • 通过观察 ARSSI 退化，MN 开始发现阶段
  • MN 恢复与服务 AP 的数据通信，直到找到更好的 AP
  • 成功切换后，执行 RPL 算法的无效过程
  • 为了评估潜在的父母，MN 广播一串 DIS 控制消息
  • 然后，所有相邻 AP 在非冲突的基础上回复 MN
  • 平均 RSSI 级别嵌入在单播 DIO 回复中。 收到每个 DIO 回复后，MN 将 ARSSI 值与 Th 电平进行比较。
    • 如果不满意（ARSSI 低于 Th），MN 将继续定期广播 DIS 突发（相对于切换定时器）
  • 在检测到高链路质量（ARSSI 高于 Th）时，发现阶段停止并且 MN 恢复常规数据

mRPL 配置和评估

为了实施和评估 mRPL，我们选择了支持 Cooja 模拟器的 Contiki 2.6.1 [36]操作系统 (OS)。选择 Contiki 的主要原因是：

  1. 相当成熟和广泛使用的 RPL/6LoWPAN 实现的可用性，
  2. 将 Cooja 代码移植到硬件平台的容易性，
  3. 移动性的可用性Cooja中的插件，可以在可重复的环境中评估 mRPL。

评估指标

Packet Delivery Ratio (PDR)

它被定义为所有 AP 成功接收的数据包数量与从 MN 发送的数据包总数之比。将 mRPL 的成功交付率与存在移动性的不同 RPL 场景进行比较。

DAO 数据包总数

为了建立下行路由，RPL 节点向上发送单播DAO 消息。

DAO 消息的下一跳目的地是首选父级。切换到最佳父节点后，子节点将其断开连接通知前一个父节点，并通知所选父节点其可达性。

DAO 数据包的总数是评估路由响应性和启用移动性的网络中的切换数量的指标。

实验配置

  • 通过提供快速可靠的切换过程来保证连接性
  • 接入点和移动节点是路由器；MN 生成数据，AP 将它们转发给根。

Timers 计时器

我们实现了四个主要计时器，以便在短时间内轻松感知链路退化和父节点不可达。 这些定时器在数据传输和发现阶段的使用在以下算法中实例化，并简要描述如下

  1. Connectivity timer (TC)
    • 移动节点不断监视信道活动以检测来自其服务 AP 的任何数据包接收
    • 每个 MN 都运行一个计时器来提高 RPL 路由响应能力——连接性计时器connectivity timer
      • 连接定时器的周期根据最大涓流间隔（Imax）设置
    • 在此期间，MN 继续监听信道并监视来自服务父节点的传入数据包
    • 在经过TC后，如果 MN 观察到一个沉默的父母，则它开始发现阶段。
    • 在检测到来自服务 AP 的任何数据包接收（例如 Trickle DIO、单播 DIO 或数据包）时，连接计时器被重置
  2. Mobility detection timer (TMD)
    • MN 的周期性 DIS 信标向服务 AP 请求单播 DIO 消息
    • MN 读取与 DIO 消息相关的 ARSSI 级别以评估链路的可靠性
      • 移动节点或在两个节点之间出现障碍可能会导致丢失请求或回复数据包
      • 在这种情况下，MN 开始发现阶段以寻找新的服务父节点。
    • 移动性检测mobility detection定时器的周期根据MN的数据生成速率设置
  3. Hand-off timer (THO)
    • 减少切换延迟hand-off delay至关重要。
    • 该定时器管理向相邻父节点突发广播 DIS 的周期。
    • 这段时间应该能够适应以尽可能高的速率传输 DIS 的突发和从相邻节点接收间歇性回复。
    • 在发送每个突发后立即收集 DIO 回复。 每个父母发送回复的顺序是这样安排的，以减少冲突的可能性
  4. Reply timer (TR)
    • 服务父母应该通过在某些时刻单播 DIO 控制消息来回复 MN
    • 选择错误的回复时间可能会导致与数据包发生冲突，进而触发发现阶段
    • 父节点从从 MN 接收到的数据包中提取相关信息（例如每个窗口大小的数据包计数器）
    • The reply time is calculated by （ws - C）×TDIS
      • C 表示每个窗口大小 (ws) 内 DIS 数据包的计数器
      • TDIS 表示 DIS 间隔
      • 该回复时间在接收到新数据包时自适应地改变

优点

  • mRPL 能够提供接近 100% 的数据包交付率。
  • mRPL 的控制消息开销与具有最小开销的 RPL 设置相当。
  • mRPL 对网络动态非常敏感。
  • mRPL 对于正常人类步行速度范围是有效的。
  • mRPL 在低流量网络中的开销较小。
  • 无论网络流量和移动节点速度如何，切换延迟都很低。

缺点

网络连通性

RPL 在网络连通性方面的主要缺点是：

  1. 如果网络已经稳定了一段时间，则无法快速检测到由于节点移动性引起的突然变化，
  2. ND 协议是在父端发起的（就像被动的切换），这会扩大切换持续时间，
  3. 恢复 ND 协议（重建路由树）非常昂贵。

实验发现

  • 增加收听周期会降低切换性能。

  • 在随机移动模式场景中，切换延迟很低。

  • 网络密度对网络开销有直接影响。

  • 成功的父切换基于数据速率和涓流设置。

  • mRPL 性能与 Trickle 设置无关。

  • 无线链路的更高重叠增加了分组传送率。

  • 更高的链路重叠减少了父交换的可能性。
