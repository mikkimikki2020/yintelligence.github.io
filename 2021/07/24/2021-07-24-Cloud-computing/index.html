1. Characteristics

The National Institute of Standards and Technology (NIST) lists five essential characteristics:

  1. On-demand self-service
  2. Broad network access.
  3. Resource pooling.
  4. Rapid elasticity.
  5. Measured service.

2. Services

Types of cloud services:

  • Software as a Service (SaaS)
  • Platform as a Service (PaaS)
  • Infrastructure as a Service (IaaS)

2.1 Software as a Service (SaaS)

Cloud-based software solutions without managing hardware, software, OS etc.

  • Simple operation (browser)
  • Pay per use or per user
  • Scalability
  • Security
  • Reliability
  • High availability

Examples:

  • Google Docs
  • Microsoft Office 365
  • Financial Planning
  • Human Resources
  • ...

2.2 Platform as a Service (PaaS)

Platforms for system administrators and developers to build, test and deploy custom applications

  • Expensive cost management system

Popular services

  • Storage
  • Database
  • Scalability

Examples:

  • Google App Engine
  • Amazon Web Services - Simple Storage Service (AWS S3)
  • Microsoft Azure services

2.3 Infrastructure as a Service (IaaS)

Computer infrastructure such as virtualization, raw storage, and networking that offers scalability.

Infrastructure stack consists of 3 essential building blocks:

  • Compute: Scalable VM resources!
  • Networking: Networking between VMs and public access with firewall control & load balancing
  • Storage: Persistent networked Block and Object storage

Examples:

  • Amazon Web Services (AWS) – e.g. Elastic Compute Cloud (EC2)

  • Microsoft Azure

  • Google Cloud

  • OpenStack

    Public Cloud Services Comparison

3. Virtualization in cloud computing

Cloud computing takes virtualization one step further:

  • No need to physically own the hardware

    无需实际拥有硬件

    • No up-front investment cost

      无前期投资成本

  • Resources are rented from the cloud provider for a fixed or dynamic time period

    在固定或动态的时间段内从云提供商处租用资源

    • Allows getting billed only for what is used

      只允许对使用的内容收费

  • Cloud providers allow creating virtual machines:

    云提供商允许创建虚拟机

    • Choose the OS and software each instance will have by deploying a VM image

      通过部署 VM 映像选择每个实例将拥有的操作系统和软件

      • These images can be prebuilt with all software already included, or require some additional work

        这些映像可以使用已包含的所有软件进行预构建，或者需要一些额外的工作

    • Scalability : Instantiate additional virtual servers or shut down existing ones within minutes

      可扩展性：在几分钟内实例化额外的虚拟服务器或关闭现有的

3.1 Private, public & hybrid cloud

What is private cloud?

  • Cloud infrastructure deployed on dedicated machines, often on-premise.

Why on-premise private cloud?

  • Security, privacy, latency, complete infrastructure control, cost (if well dimensioned & in-house knowledge)
  • Keep sensitive data within the premises

Why public cloud?

  • Low/no up-front investment costs, virtually endless scalability

  • 99,9% SLA uptime, multiple availability zones

    → Shared cloud infrastructure

Hybrid cloud

  • A combination of on-premise private cloud and public cloud

4. OpenStack

4.0 What is OpenStack?

Open Source Infrastructure as a Service (IaaS) cloud platform

  • Basic building blocks: Compute, Storage, Networking, Authentication
  • Various optional components: orchestration, provisioning, metering, etc.

4.1 Why use OpenStack?

  • Avoid vender lock-in from the major public cloud provider platforms

    避免主要公共云提供商平台的供应商锁定

  • Complete control over on-premise cloud infrastructure

    完全控制本地云基础设施

  • Keep sensitive data within the premises

    将敏感数据保存在场所内

4.2 Components

4.3 Deployment

  • OpenStack Victoria (10/2020) operates on top of a Linux host operating system

    • RHEL / CentOS 8, Ubuntu 20.04 focal fossa, Debian 10 buster
    • OpenStack Nova: Virtualization via QEMU with KVM backbone
    • OpenStack components: open-source and written in Python

  • Various OpenStack deployment methods and tools available

    • We want harmonized & automated way to deploy & maintain OpenStack components over multiple machines

    • OpenStack Kolla project

      • Containerized versions of OpenStack components
      • Well tested & mostly independent of host OS configuration

    • Ansible: automated software provisioning & deployments

      => Kolla-Ansible deployment scripts

4.4 Next generation

  • Shift from monolithic applications to microservices
  • Shift from virtual machines to containers
    • Docker Containerization

4.5 Deploy a simple infrastructure on OpenStack 实践部署

  1. Create a private network
  2. Define a subnet
  3. Run a VM
  4. Connect it to the subnet
  5. Create a block storage
  6. Attach it to the VM

5. Cloud-like container

  • Manage containers over multiple machines
  • Networking capabilities (+ DNS resolution)
  • Persistent (network) storage volumes

Scaling capabilities:

  • Horizontal scaling via replicas & load balancing
  • Vertical scaling by modifying RAM & CPU allocation

Example:

  • Kubernetes
    • Amazon ECS
    • MS Azure AKS
    • Google GKE
    • Self-hosted (e.g. minikube)
  • Docker Swarm
  • HashiCorp Nomad
  • Apache Mesos

5.1 Kubernetes

5.1.1 POD

Kubernetes PODs are the smallest building blocks. They can contain:

  • Compute
    • Information about the containerized application(s)
  • Storage
    • Volumes to be mounted
  • Network
    • Which ports should be accessible on the pod’s bridged IP address

5.1.2 NODE

Kubernetes NODEs are the machines that run the Kubernetes environment

  • Nodes run a variety of pods
  • Can be a physical or virtual machine

Worker nodes run kubelet and docker (or other container engine)

  • They take orders from a Kubernetes control-plane nodes

Control-plane nodes control and schedule the Kubernetes cluster

  • In case of multiple control-plane nodes providing High-Availability:
    • A consensus quorum between control-plane nodes is required for making decisions
      • More than half of the nodes must be available and agree

5.1.3 DEPLOYMENT - update pod 更新

Kubernetes DEPLOYMENTs are used to describe and update (a collection of) pods and how many replicas get deployed and maintained.

  • Deployments orchestrate the described pods which provide the applications and services
  • Pods are automatically redeployed (one by one) in case of an update or failure

Examples:

  • We want the specified type of pod automatically redeployed in case the existing pod terminates/crashes

    我们希望在现有 pod 终止/崩溃的情况下自动重新部署指定类型的 pod

  • We want to specify the number of replicas of a certain pod

    我们要指定某个 pod 的副本数

  • We want to deploy x replicas of type A pods and y replicas of type B pods

    我们要部署 A 类 Pod 的 x个副本和 B 类 Pod 的 y 个副本

  • We want to update the environment variables or mount point of a certain type of pod and have them redeployed

    我们想要更新某种类型 pod 的环境变量或挂载点并重新部署它们

5.1.4 AUTOSCALER

An AUTOSCALER can be added to dynamically scale replicas based on monitored resource consumption

可以添加 AUTOSCALER 以根据受监控的资源消耗动态扩展副本

5.1.5 SERVICE - Networking service 网络

Kubernetes SERVICEs are means to expose (a set of) pods as a networking service

Kubernetes SERVICEs 是将（一组）pods 作为网络服务公开的手段

Why do we need services?

  • Pods are often short-lived entities that each have a dynamic IP address

    Pod 通常是短期实体，每个实体都有一个动态 IP 地址

    • We want a long-lived entry-point IP address to communicate over

      我们想要一个长期存在的入口点 IP 地址来进行通信

  • How to get a ‘front-end’ application communicating with these ‘back-end’ pods?

    如何让“前端”应用程序与这些“后端”pod 通信？

    • How to specify which replica pod to communicate with?

      如何指定要与哪个副本 pod 通信？

    • On which node is this pod running?

      这个 Pod 在哪个节点上运行？

Solution:

Services provide an abstracted and unified ‘front-end’ entry-point to the (replicated) ‘back-end’ pods

服务为（复制的）“后端”pod 提供了一个抽象且统一的“前端”入口点

  • Round-robin is used as algorithm to distribute load between the ‘back-end’ pods

    Round-robin被用作在“后端”pod 之间分配负载的算法

  • Several types of services with different purposes:

    • ClusterIP

      • provides a cluster-internal IP address for internal communication

        为内部通信提供集群内部 IP 地址

    • NodePort

      • externally exposes the service on all nodes via port mapping.

        通过端口映射对外暴露所有节点上的服务

      • External access via : <nodeIP>: <nodePort>

        通过以下方式进行外部访问：<nodeIP>:<nodePort>

    • LoadBalancer

      • externally exposes the service via a new cluster-external IP address

        通过新的集群外部 IP 地址向外部公开服务

5.1.6 PersistentVolume (PV) -link storage 连接存储

PersistentVolume (PV) links Kubernetes to a physical storage mount

5.1.7 PersistentVolumeClaims (PVC)

PersistentVolumeClaims (PVC) are used to request a persistentVolume

  • The PVC binds to a PV. Other PVC can no longer bind to this PV.
  • Multiple (types of) pods can mount the PVC

5.1.8 Namespaces- create environments 创造环境

Namespaces are used to create separate environments

  • For different applications/deployments
  • For different users

5.1.9 Authentication

  • Authentication via ServiceAccountor externally managed (e.g. X509 client certificate)
    • ServiceAccounts have a Secret Token for authentication purposes
  • Authorization via Role-Based Access Control (RBAC)
    • Roles (inside a Namespace)
      • + RoleBinding
    • ClusterRole (inside the entire cluster)
      • + ClusterRoleBinding

5.2 Which one suits?

There is no distinct answer… It depends (among others) on:

  • Design of the application
    • Large monolithic
      • → virtual machines
    • Microservices
      • → containers
  • Application longevity 应用寿命
    • Stateful & persistent apps
      • → virtual machines
    • Stateless & short-lived apps
      • → containers
  • Scalability 可扩展性
    • → containers
  • Isolation requirements
    • → virtual machines
  • Application compatibility
    • virtual machines (e.g. windows-only application)

6. Managing clouds 云管理

6.1 Infrastructure as a Service (IaaS)

  • Cloud Management is the management of cloud computing products and services.

    云管理是对云计算产品和服务的管理。

  • Cloud Management Software is an integrated product that helps in the management of cloud environment with the sole aim of providing agility and cost saving.

    云管理软件是一种集成产品，可帮助管理云环境，其唯一目标是提供敏捷性和成本节约。

6.1.1 Objects

  • Simplify complexity
  • File storage and backup
  • Automation
  • Build for the future
  • Self Service
  • Easy-to-use client interface
  • Effective and efficient management of cloud infrastructure
  • Scalable deployment
  • Robust performances

6.2 Terraform

  • Opensource Infrastructure-as-Code software tool
  • Efficient deployment, management and automation
  • Compatible with 500+ providers (public/private clouds, network appliances, Platform as a Service, Software as a Service)
  • Constantly updated and supported by Hashicorp and the community
  • Part of Hashicorp's “Cloud-oriented” suite of tools

6.2.1 Why Terraform?

  • One tool to manage any resource, regardless of where
  • Excellent to handle multi-cloud / hybrid cloud scenarios, but not only
  • Deploy, manage and update with Infrastructure as Code

6.2.2 Infrastructure-as-Code (IaC)

6.2.3 Advantages

  • Speed up the deployment process
  • Reusability
  • Automate process
  • Less human errors
  • Better view on the infrastructure
  • Testable
  • Version control systems (GitHub)
  • Best practices

6.2.4 Architecture

6.2.5 Core main commands

  • INIT
    • Initialize Terraform and look for providers
  • PLAN
    • Overview of what to execute to realize what is described in the configuration files
  • APPLY
    • Perform the operations as planned
  • DESTROY
    • Deallocate and destroy all the resources

  • Terraform Cloud

7. Automating VM image generation 自动生成镜像

Cloud deployments benefit from pre-baked images (e.g. with integrated GPU drivers)

  • Creating & modifying VM/container images is a repetitive and time-consuming task
  • There is a need for automated image creation
    • e.g. Triggered each time a new GIT commit is successfully built and ready for deployment (CI/CD)

7.1 Hashicorp packer

Hashicorp packer is a multi-platform tool which provides this automation

  • Packer binary
  • Image build template in Hashicorp Configuration Language (HCL)
    • Builders
      • EC2, Azure, Google Cloud, DigitalOcean, Docker, OpenStack, QEMU, VMware, …
    • Provisioners
      • File, Shell, Ansible, Chef, Puppet, …

OpenStack environment variables (credentials etc.) loaded separately

packer build centos-gpu.hcl
