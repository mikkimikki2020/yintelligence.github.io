1. Definition 定义

“Virtualization is the ability to run multiple operating systems on a single physical system and share the underlying hardware resources"

“虚拟化是在单个物理系统上运行多个操作系统并共享底层硬件资源的能力。”

Virtualization refers to the act of creating a virtual (rather than actual) version of something, including virtual computer hardware platforms, computer network resources, etc.

虚拟化是指创建某物的虚拟（而非实际）版本的行为，包括虚拟计算机硬件平台、计算机网络资源等。

1.0 Motivation 动机

Three fundamental abstractions are necessary to describe the operation of a computing systems:

  1. processors
  2. memory
  3. communications

As the scale of a system and the size of its users grows, it becomes very challenging to manage its recourses (see three points above)

随着系统规模和用户规模的增长，管理其资源变得非常具有挑战性（参见以上三点）

Resource management issues:

  • provision for peak demands → overprovisioning

    提供高峰需求 → 过度供应

  • heterogeneity of hardware and software 硬件和软件的异构性

  • machine failures 机器故障

Virtualization is a basic enabler of Cloud Computing, it simplifies the management of physical resources for the three abstractions

虚拟化是云计算的基本推动者，它简化了三个抽象的物理资源管理

  • For example, the state of a virtual machine (VM) running under a virtual machine monitor (VMM) can de saved and migrated to another server to balance the load

    VMM 也称为管理程序hypervisor。

    例如，在虚拟机监视器（VMM）下运行的虚拟机（VM）的状态可以去保存并迁移到另一台服务器以平衡负载

  • For example, virtualization allows users to operate in environments they are familiar with, rather than forcing them to specific ones

    例如，虚拟化允许用户在他们熟悉的环境中操作，而不是强迫他们使用特定的环境

Virtualization is important for 虚拟化对于以下方面很重要:

  • Performance isolation 性能隔离

    • as we can dynamically assign and account for resources across different applications

      因为我们可以跨不同应用程序动态分配和说明资源

  • System security 系统安全

    • as it allows isolation of services running on the same hardware

      因为它允许隔离在同一硬件上运行的服务

  • Performance and reliability 性能和可靠性

    • Reliability

      • By putting each service on a separate computer, if one of the servers crashes, at least the other ones are not affected

        通过将每个服务放在单独的计算机上，如果其中一个服务器崩溃，至少其他服务器不受影响

    • Security

      • Malevolent intruders affecting one service can be isolated from others (sandboxing concept)

        影响一项服务的恶意入侵者可以与其他服务隔离（沙盒sandboxing概念）

    • as it allows applications to migrate from one platform to another

      因为它允许应用程序从一个平台迁移到另一个平台

  • The development and management of services offered by a provider

    提供商提供的服务的开发和管理

1.1 Advantages 优点

  • Multi O.S. support 多系统支持

    • e.g. a Web server on Linux, a mail server on Windows, an ecommerce server for customers running on OS X

  • Isolation 隔离性

    • Virtual machines are isolated from each other as if they are physically separated

      虚拟机彼此隔离，就好像它们在物理上是分开的

  • Economical 经济性

    • Sharing of resources helps cost reduction

      资源共享有助于降低成本

    • fewer physical machines saves money on hardware and electricity and takes up less rack space.

      更少的物理机可以节省硬件和电力，占用更少的机架空间。

  • Encapsulation 封装性

    • Virtual machines encapsulate a complete computing environment

      虚拟机封装了一个完整的计算环境

    • experimentation, evaluation of security threads,…

      实验，安全线程的评估，...

  • Hardware Independence 硬件独立

    • Virtual machines run independently of underlying hardware

      虚拟机独立于底层硬件运行

  • Portability 便携性

    • Virtual machines can be migrated between different hosts.

      虚拟机可以在不同主机之间迁移。

2 History 历史

  • Popular research idea in 60s and 70s
    • Entire conferences on virtual machine monitor
    • Hardware/VMM/OS designed together
  • 60s: IBM CP/CMS control program, a virtual machine operating system for the IBM System/360 Model 67
    • Run different OS instances
    • Desirable when few machines is around
  • 70s: Popek and Golberg from UCLA published “Formal Requirements for Virtualizable Third Generation Architectures” in 1974
    • List of conditions that a computer architecture should satisfy to support virtualization efficiently.
    • The popular x86 architecture that originated in the 1970s did not support these requirements for decades.
  • Interest died out in the 80s
  • 90’s
    • VMware: Researchers developed a new hypervisor and founded VMware, the biggest virtualization company of today’s. First virtualization solution was is 1999 for x86.
    • Disco: Stanford research project (SOSP’97)
      • By Edouard Bugnion, Scott Devine, and Mendel Rosenblum
        • Rosenblum is the author of log-structured file system
      • So OS developers won’t change their code for new hardware
  • 2000’s
    • IBM: z-series with 64-bit virtual address spaces and backward compatible with the System/360
    • Research virtual machines: Xen (SOSP’03)
    • Hardware support: Intel VT-x and AMD’s AMD-V (2005)
  • Today many virtualization solutions: Xen from Cambridge, KVM, Hyper-V, …
  • IBM was the first to produce and sell virtualization for the mainframe. But, VMware popularised virtualization for the masses.

2 Implementation 实现

Virtualization simulates the interface to a physical object by:

虚拟化通过以下方式模拟物理对象的接口：

  • Multiplexing 多路复用

    • creates multiple virtual objects from one instance of a physical object.

      Many virtual objects to one physical.

      从一个物理对象的一个实例创建多个虚拟对象。

      许多虚拟对象到一个物理对象。

    • Example - a processor is multiplexed among a number of processes or threads.

      示例 - 处理器在多个进程或线程之间多路复用。

  • Aggregation 聚合

    • creates one virtual object from multiple physical objects.

      One virtual object to many physical objects.

      从多个物理对象创建一个虚拟对象。

      一个虚拟对象到多个物理对象。

    • Example - a number of physical disks are aggregated into a RAID disk.

      示例 - 多个物理磁盘聚合为一个 RAID 磁盘。

  • Emulation 仿真

    • constructs a virtual object of a certain type from a different type of a physical object.

      从不同类型的物理对象构造特定类型的虚拟对象。

    • Example - a physical disk emulates a Random Access Memory (RAM).

      物理磁盘模拟随机存取存储器 (RAM)。

  • Multiplexing and emulation 多路复用和仿真

    • Examples

      • virtual memory with paging multiplexes real memory and disk

        带分页的虚拟内存多路复用实内存和磁盘

      • a virtual address emulates a real address.

        虚拟地址模拟真实地址

3. Requirements 需求 - Hypervisors

The main idea is that a VMM (Virtual Machine Monitor) creates the illusion of multiple (virtual) machines on the same physical hardware. A VMM is also known as a hypervisor.

主要思想是 VMM（虚拟机监视器）在同一物理硬件上创建多个（虚拟）机器的错觉。 VMM 也称为管理程序hypervisor。

Hypervisors should score well in three dimensions:

  1. Safety 安全

    • the hypervisor should have full control of the virtualized resources.

      管理程序应该完全控制虚拟化资源。

  2. Fidelity 保真度

    • the behaviour of a program on a virtual machine should be identical to that of the same program running on bare hardware.

      一个程序在虚拟机上的行为应该与在裸硬件上运行的同一个程序的行为相同。

  3. Efficiency 效率

    • much of the code in the virtual machine should run without intervention by the hypervisor.

      虚拟机中的大部分代码应该在没有管理程序干预的情况下运行。

Every CPU with kernel mode and user mode has a set of instructions that behave differently when executed in kernel mode than when executed in user mode.

每个具有内核模式和用户模式的 CPU 都有一组指令，这些指令在内核模式下执行时的行为与在用户模式下执行时的行为不同。

4.1 Full virtualization 完全虚拟化

  • The result is a system in which all software (including all OS’s) capable of execution on the raw hardware can be run in the virtual machine.

    结果是一个能够在原始硬件上执行的所有软件（包括所有操作系统）都可以在虚拟机中运行的系统。

  • Comprehensively simulate all computing elements as instruction set, main memory, interrupts, exceptions, and device access.

    全面模拟指令集、主存、中断、异常、设备访问等所有计算元素。

  • Full virtualization is only possible given the right combination of hardware and software elements.

    只有在硬件和软件元素的正确组合下才能实现完全虚拟化。

  • Allows the sharing a computer system among multiple users

    允许多个用户共享一个计算机系统

    • Isolating users from each other (and from the control program)

      将用户彼此隔离（以及与控制程序）

    • Emulating new hardware to achieve improved reliability, security and productivity

      模拟新硬件以提高可靠性、安全性和生产力

  • It needs a single machine that could be multiplexed among many users.

    Each such virtual machine had the complete capabilities of the underlying machine, and (for its user) the virtual machine was indistinguishable from a private system.

    它需要一台可以在许多用户之间复用的机器。

    每个这样的虚拟机都具有底层机器的完整功能，并且（对于它的用户）虚拟机与私有系统没有区别。

4.1.1 Examples 例子

  • First demonstrated with IBM's CP-40 research system in 1967
  • Re-implemented CP/CMS in IBM's VM family from 1972 to the present.
  • Each CP/CMS user was provided a simulated, stand-alone computer.

4.1.2 Challenge 挑战

不是所有指令都可以直接运行；不是所有硬件都适合完全虚拟化；同时也要考虑到安全和IO操作问题。

4.1.2.1 Software

The effects of every operation performed within a given virtual machine must be kept within that virtual machine – virtual operations cannot be allowed to alter the state of any other virtual machine, the control program, or the hardware

在给定虚拟机内执行的每个操作的效果必须保留在该虚拟机内——不允许虚拟操作改变任何其他虚拟机、控制程序或硬件的状态

  • Security issues 安全问题 – Interception 拦截
  • Simulation of privileged operations 模拟特权操作 -- I/O instructions
4.1.2.2 Hardware

  • Some machine instructions can be executed directly by the hardware

    有些机器指令可以由硬件直接执行，

    • E.g., memory locations and arithmetic registers.

      例如，内存位置和算术寄存器。

  • But other instructions that would "pierce the virtual machine" cannot be allowed to execute directly

    但是其他会“刺穿虚拟机pierce the virtual machine”的指令不能直接执行

    • they must instead be trapped and simulated.

      相反，他们必须被困住和模拟。

    • Such instructions either access or affect state information that is outside the virtual machine.

      此类指令访问或影响虚拟机外部的状态信息。

  • Some hardware is not easy to be used for full virtualization, e.g., x86

    一些硬件不容易用于完全虚拟化，例如，x86

4.2 Binary translation 二进制翻译

Used by VMware at one point

Binary translation rewrites parts of the code on the fly to replace sensitive but not privileged instructions with safe code to emulate the original instruction

二进制翻译即时重写部分代码，用安全代码替换敏感但非特权指令以模拟原始指令

4.2.1 Steps 步骤

Replace problematic calls dynamically:

动态替换有问题的调用：

  1. Read in code, looking for basic blocks

    读入代码，寻找基本块

  2. Then inspect basic block to find problematic instructions. If found, replace with VM call (process called binary translation).

    然后检查基本块以查找有问题的指令。 如果找到，则替换为 VM 调用（称为二进制转换的过程）

  3. Then, cache block and execute.

    然后，缓存块并执行

  4. Eventually, most basic blocks will not be modified and cached, and will run at near native speed.

    最终，大多数基本块将不会被修改和缓存，并将以接近本机的速度运行。

  5. Can force traps on sensitive non-privileged instructions

    可以对敏感的非特权指令强制陷阱

4.2.2 Challenge 挑战

Imperfect hardware (x86) doesn’t allow perfect trap-and-emulate

  • popf (pop flags)

    • might modify interrupt flag (IF) when executed in kernel mode, but suppress the attempt to modify IF when executed in user mode

      在内核模式下执行时可能会修改中断标志 (IF)，但在用户模式下执行时禁止尝试修改 IF

  • Code segment selector (%cs) have two bits storing the kernel/user mode, can be read by user mode

    代码段选择器（%cs）有两位存储内核/用户模式，可以被用户模式读取

No longer used today because of more powerful H/W support

4.2.3 Advantages

  • No hardware assistance,
  • No modifications of the guest OS
  • Isolation, Security

4.2.4 Disadvantages

  • Speed of execution
  • Portability

4.3 Paravirtualization 半虚拟化

It is very difficult to build sophisticated binary translation support necessary for full virtualization.

构建完全虚拟化所需的复杂二进制转换支持非常困难。

Examples: Xen, Denali

A recent approach (paravirtualization) is to modify the source code of the guest OS such that it makes hypervisor calls instead of executing “privileged instructions”.

最近的一种方法（半虚拟化）是修改来宾操作系统的源代码，使其调用管理程序hypervisor而不是执行“特权指令”。

  • Paravirtualization does not aims to present a virtual machine that looks just like the actual underlying hardware

    半虚拟化的目的不是提供看起来像实际底层硬件的虚拟机

  • Paravirtualization involves modifying the OS kernel to replace non-virtualizable instructions with hypercells that communicate directly with the virtualization layer hypervisor.

  • 半虚拟化涉及修改操作系统内核，用与虚拟化层管理程序直接通信的超级调用替换不可虚拟化的指令。

  • Much easier to modify source code than to emulate hardware instructions (as in binary translation).

    修改源代码比模拟硬件指令要容易得多（如在二进制翻译中）。

  • Paravirtualization cannot support unmodified OS

    半虚拟化不能支持未修改的操作系统

    • Paravirtualization is different from full virtualization, where the unmodified OS does not know it is virtualized and sensitive OS calls are trapped using binary translation.

      半虚拟化与完全虚拟化不同，其中未修改的操作系统不知道它已虚拟化，敏感的操作系统调用使用二进制转换被捕获。

    • Both type 1 and type 2 hypervisors work with unmodified guest OS’s:

      见6.3 Hypervisor - Approach

      • Avoid calls to non-privileged sensitive instructions (changed to hypervisor calls).

        避免调用非特权敏感指令（更改为管理程序调用）。

      • Can include optimizations.

        可以包括优化。

      • Use real hardware resources.

        使用真实的硬件资源。

4.3.1 Hypercalls

Present machine-like software interface that explicitly exposes the fact that it is a virtualized environment:

呈现类似机器的软件界面，明确暴露了它是一个虚拟化环境的事实:

  • Hypercalls

    • allow the guest to send explicit requests to the hypervisor (much as a system call offers kernel services to applications).

      允许来宾向管理程序发送显式请求（就像系统调用向应用程序提供内核服务一样）。

    • provides hypercall interfaces for other critical kernel operations such as memory management, interrupt handling and time keeping.

      为其他关键内核操作（如内存管理、中断处理和计时）提供超级调用接口。

    • are used for privileged sensitive operations like updating the page tables, but because they do it explicitly in cooperation with the hypervisor, the overall system can be simpler and faster

      用于像更新页表这样的特权敏感操作，但因为它们明确地与管理程序合作，所以整个系统可以更简单、更快

For this the hypervisors have to define an API, as an interface to the guest OS’s.

  • But this approach is similar to the microkernel approach.

    但这种方法类似于微内核方法。

  • A guest OS whose “privileged instructions” are replaced with hypervisor calls is said to be para-virtualized.

    将“特权指令”替换为管理程序调用的来宾操作系统称为半虚拟化。

4.3.2 Advantage

  • faster execution
  • lower virtualization overhead

4.3.3 Disadvantage

  • poor portability
  • Para-virtualized systems won’t run on native hardware (needs hypervisor).

4.4 Hardware Assisted Virtualization (future direction)

Examples: Intel VT-x, Xen 3.x

  • Hardware switch supported by CPU.

    CPU 支持的硬件切换

  • Also known as:

    • accelerated virtualization 加速虚拟化,
    • hardware virtual machine 硬件虚拟机 (Xen)
    • native virtualization 原生虚拟化 (Virtual iron)

A new CPU execution mode feature that allows the VMM to run in a new root mode below ring 0.

新的 CPU 执行模式功能，允许 VMM 在环 0 以下的新根模式下运行。

As depicted in figure, privileged and sensitive calls are set to automatically trap to the hypervisor, removing the need for either binary translation or paravirtualization.

如图所示，特权和敏感调用设置为自动捕获到管理程序，无需二进制转换或半虚拟化

  • The guest state is stored in Virtual Machine Control Structures (VT-x) or Virtual Machine Control Blocks (AMD-V).

    来宾状态存储在虚拟机控制结构 (VT-x) 或虚拟机控制块 (AMD-V) 中。

  • High hypervisor to guest transition overhead and a rigid programming model

    高管理程序到客户转换开销和严格的编程模型

  • Even with VM hardware support binary translation can still be used to improve performance.

    即使有 VM 硬件支持，二进制转换仍然可以用来提高性能。

4.4.1 Implement - VMCS

AMD SVM (Secure Virtual Machine) and Intel VT (Virtualization Technology) CPU’s have created virtualization support after 2005

AMD SVM（安全虚拟机）和 Intel VT（虚拟化技术）CPU 在 2005 年之后创建了虚拟化支持

  • New assembly commands to enter VM mode

    用于进入 VM 模式的新程序集命令

  • Hypervisor (VMM) runs on ring 0 under root mode

    管理程序 (VMM) 在 root 模式下在环 0 上运行

  • Guest OS runs in ring 0 under non-root mode

    来宾操作系统在非 root 模式下在 ring 0 中运行

  • Changes are done within VM specific state called VMCS (Virtual Machine Control Structure)

    更改在称为 VMCS（虚拟机控制结构）的 VM 特定状态内完成

4.4.2 Advantage

Even faster execution

4.4.3 Intel VT-X

In 2005 Intel released two Pentium 4 models supporting VT-x.

VT-x supports two modes of operations:

  • VMX root - for VMM operations.
  • VMX non-root - support a VM.

And a new data structure called the Virtual Machine Control Structure including host-state and guest-state area.

以及一种称为虚拟机控制结构的新数据结构，包括主机状态和来宾状态区域。

  • VM entry

    • the processor state is loaded from the guest-state of the VM scheduled to run; then the control is transferred from VMM to the VM.

      处理器状态从预定运行的虚拟机的来宾状态加载； 然后控制权从 VMM 转移到 VM

  • VM exit

    • saves the processor state in the guest-state area of the running VM; then it loads the processor state from the host-state area, finally transfers control to the VMM.

      将处理器状态保存在正在运行的 VM 的来宾状态区域中； 然后它从主机状态区加载处理器状态，最后将控制权转移到 VMM。

5. Virtualization of the CPU

5.0 (Recap) Kernel mode vs. User Mode

  • Dual-mode operation allows OS to protect itself and other system components

    双模式操作允许操作系统保护自身和其他系统组件

    • User mode
    • Kernel mode

  • Mode bit provided by hardware

    硬件提供的模式位

    • Ability to distinguish when system is running user or kernel code

      能够区分系统何时运行用户或内核代码

    • Some instructions are privileged, only executable in kernel mode

      一些指令是特权指令，只能在内核模式下执行

    • System call changes mode to kernel, return resets it to user

      系统调用将模式更改为内核，返回将其重置为用户

5.0.1 Hardware-enforced protection mechanism 硬件强制保护机制

  • Kernel-code (in particular, interrupt handlers) runs in kernel mode

    内核代码（特别是中断处理程序IH）在内核模式下运行

    • the hardware allows all machine instructions to be executed and allows unrestricted access to memory and I/O ports

      硬件允许执行所有机器指令，并允许不受限制地访问内存和 I/O 端口

  • Everything else runs in user mode 其他一切都在用户模式下运行

  • Therefore, since our VM does not run in kernel mode, we cannot run sensitive instructions in it, since those must trap to kernel space.

    因此，由于我们的 VM 不在内核模式下运行，我们不能在其中运行敏感指令，因为这些指令必须捕获到内核空间。

5.0.2 CPU protection ring

Four layers of privilege execution → CPU protection ring.

Normally:

  • User applications run in ring 3

    用户应用程序在环 3 中运行

  • OS runs in ring 0

    操作系统在环 0 中运行

In which ring should the VMM run?

VMM 应该在哪个环中运行？

  • In ring 0, then, same privileges as an OS → wrong

    在环 0 中，与操作系统相同的权限 → 错误

  • In rings 1,2,3, then OS has higher privileges → wrong

    在环 1,2,3 中，则操作系统具有更高的权限 → 错误

  • Move the OS to ring 1 and the VMM in ring 0 → OK

    将操作系统移至环 1 并将 VMM 移至环 0 → OK

Supervisor mode

  • User programs use ring 3 用户程序使用环 3

  • Hypervisor runs on ring 0

    • Can run any instruction (ring 0).

      可以运行任何指令（环 0）。

    • Trusted to not fail, in case of failure the system crashes.

      万一出现故障系统崩溃被认为不会失败。

  • Guest OS runs on ring 1

5.0.3 Machine instructions 机器指令

  • Privileged instructions 特权指令

    • can be executed in kernel mode.

      可以在内核模式下执行。

    • When attempted to be executed in user mode, they cause a trap and so executed in kernel mode.

      当试图在用户模式下执行时，它们会导致陷阱并因此在内核模式下执行。

  • Non-privileged instructions 非特权指令

    • the ones that can be executed in user mode

      可以在用户模式下执行的指令

  • Assembly sensitive instructions 组装敏感指令

    • I/O, MMU control, etc.

    • can be executed in either kernel or user but they behave differently.

      可以在内核或用户中执行，但它们的行为不同。

      Sensitive instructions require special precautions at execution time.

      敏感指令在执行时需要特殊的预防措施。

Sensitive and non-privileged instructions are hard to virtualize

敏感和非特权指令难以虚拟化

A machine is virtualizable only if the sensitive instructions are a subset of the privileged instructions.

仅当敏感指令是特权指令的子集时，机器才可虚拟化。

5.1 Trap-and-emulate approach 捕获模拟方法

Trap-and-emulate the “privileged instructions” in the user mode

在用户模式下捕获Trap并模拟emulate“特权指令privileged instructions”

  • Hypervisor responsible to assist with sensitive instructions.

    Hypervisor负责协助处理敏感指令

  • typically related to kernel functions, such as I/O instructions or instructions that changes MMU settings

    通常与内核功能相关，例如 I/O 指令或更改 MMU 设置的指令

  • emulate them within the guest OS

    在Guest操作系统中模拟它们

  • H/W support (kernel + user mode)

    硬件支持（内核 + 用户模式）

  • Modern CPUs have virtualization support

    现代 CPU 具有虚拟化支持

    • 3 modes: kernel, supervisor, user

Trap-and-emulate approach:

  • Containers in which virtual machines can run

    可以运行虚拟机的容器

  • The program runs until it creates a trap

    程序运行直到它创建一个陷阱

  • Traps handled by the hypervisor to emulate the desired behaviour

    由管理程序处理的陷阱以模拟所需的行为

6.1 Memory virtualization 内存虚拟化

How to integrate the paging of the host OS with the paging of the guest OS

如何将主机操作系统的分页与来宾操作系统的分页结合起来

6.1.1 Challenge

  • Not only virtual memory 不仅是虚拟内存

  • To run multiple virtual machines on a single system, another level of memory virtualization is required.

    要在单个系统上运行多个虚拟机，需要另一个级别的内存虚拟化

  • OS tracks mapping of virtual memory pages to physical memory page frames.

    操作系统跟踪虚拟内存页面到物理内存页面框架的映射

    • But VM’s do not share mapping information.

      但是 VM 不共享映射信息

  • Can cause a collision upon page reference.

    可能会导致页面引用发生冲突

    • For example, VM1 maps VP12 into PP6 for some process.

      例如，对于某些进程，VM1 将 VP12 映射到 PP6

    • VM 2 maps VP7 into PP6 for some process.

      VM 2 将 VP7 映射到 PP6 以用于某些进程

    • Referencing the page is sensitive op. Who will own physical page #6?

      引用页面是敏感操作。 谁将拥有实体页面 #6？

6.1.2 Shadow pages 影子页表

  • Allow hypervisor to manage page mapping

    允许管理程序管理页面映射

    • Use Shadow Page Tables for the VMs.

      为 VM 使用影子页表

  • The hypervisor is responsible for mapping guest physical memory to the actual machine memory, and it uses shadow page tables to accelerate the mappings.

    管理程序负责将客户物理内存映射到实际机器内存，并使用影子页表来加速映射。

6.1.3 HW support for nested/extended page tables

Requires hardware support.

  • Two “CR3”s (CR3 and EPTP).

    CR3, an x86 microprocessor control register

  • MMU translates each guest mapping level

    MMU 翻译每个来宾映射级别

Extended/nested page tables are walked every time a guest physical address is accessed, including the accesses for each level of the guest’s page tables.

每次访问来宾物理地址时都会遍历扩展/嵌套页表，包括对来宾页表每个级别的访问。

  • Guest page table map 访客页面表图: Guest VA → Guest PA
  • Nested page table map 访客页面表图: Guest PA → Host PA

6.2 Device and I/O virtualization

  • VMM supports all device/IO drivers

  • Physically/virtually existed 物理/虚拟存在

  • Trap-and-emulate on every hardware access

    在每次硬件访问时进行陷阱和模拟

  • Too slow, so modify the device driver to use special “monitor call” (similar to syscall)

    太慢了，所以修改设备驱动程序使用特殊的“监视器调用”（类似于系统调用）

Each guest OS holds its own “partition分区”

  • Typically implemented as a file or region on disk.

    通常实现为磁盘上的文件或区域。

  • Hypervisor must convert guest OS address (block #) into physical address in region.

    管理程序必须将客户操作系统地址（块#）转换为区域中的物理地址。

  • May convert between storage types.

    可以在存储类型之间转换。

7. Hypervisors

7.1 CPU

The VMM/hypervisor partitions the resources of computer system into one or more virtual machines (VMs).

llows several operating systems to run concurrently on a single hardware platform

VMM/管理程序将计算机系统的资源划分为一个或多个虚拟机（VM）。

允许多个操作系统在单个硬件平台上同时运行。

  • A VM is an execution environment that runs an OS

    VM 是运行 OS 的执行环境

  • VM – an isolated environment that appears to be a whole computer, but actually only has access to a portion of the computer resources

    一个孤立的环境，看似是整台计算机，但实际上只能访问部分计算机资源

A VMM allows:

  • Multiple services to share the same platform

    多个服务共享同一个平台

  • Live migration - the movement of a server from one platform to another

    实时迁移 - 服务器从一个平台移动到另一个平台

  • System modification while maintaining backward compatibility with the original system

    系统修改同时保持与原系统的向后兼容性

  • Enforces isolation among the systems, thus security

    在系统之间强制隔离，从而确保安全

A guest operating system is an OS that runs in a VM under the control of the VMM.

7.2 Layers of privilege execution 权限执行

引用5.0.2 CPU protection ring

Supervisor mode

  • User programs use ring 3 用户程序使用环 3

  • Hypervisor runs on ring 0

    • Can run any instruction (ring 0).

      可以运行任何指令（环 0）。

    • Trusted to not fail, in case of failure the system crashes.

      万一出现故障系统崩溃被认为不会失败。

  • Guest OS runs on ring 1

7.3 Approach 方法

Type 1 hypervisors run on the bare metal

Type 2 hypervisors use the services of an existing host operating system.

两种实现方式

  • Type 1 hypervisor - kernel level

    • Virtualization is implemented as part of the hosting OS at the kernel level.

      虚拟化在内核级别作为托管操作系统的一部分实施。

    • Support multiple copies of the actual machine, called virtual machines.

      支持实际机器的多个副本，称为虚拟机。

  • Type 2 hypervisor - user level

    • Virtualization is implemented as a user program running at the user level.

      虚拟化作为在用户级别运行的用户程序来实现。

    • It “interprets” the machine’s instruction set which also creates a virtual machine.

      它“解释”机器的指令集，这也创建了一个虚拟机。

7.3.1 Type 1 hypervisor

  • The hypervisor runs on the bare hardware as part of the OS.

    管理程序作为操作系统的一部分在裸硬件上运行。

  • The virtual machine runs as a user process in user mode.

    虚拟机在用户模式下作为用户进程运行。

    • Is not allowed to execute “privileged instructions”.
    • 不允许执行“特权指令”。

  • The guest OS thinks it is running in kernel mode, although it is in user mode. (virtual kernel mode)

    来宾操作系统认为它在内核模式下运行，尽管它是在用户模式下。 （虚拟内核模式）

    • When the guest OS executes a “privileged instruction” , it generates a trap (thanks to the VT support from hardware) in the host OS kernel.

      当客户操作系统执行“特权指令privileged instruction”时，它会在主机操作系统内核中生成一个陷阱（感谢硬件的 VT 支持）。

    • The hypervisor inspects the instruction if it was issued by the guest OS running in the virtual machine.

      管理程序hypervisor检查指令是否由在虚拟机中运行的来宾操作系统发出。

      • If so, the hypervisor emulates what the real hardware would do when confronted with that “privileged instruction” executed in user mode

        如果是这样，当遇到在用户模式下执行的“特权指令”时，虚拟机管理程序会模拟真实硬件会做什么

7.3.2 Type 2 hypervisor

  • The hypervisor runs as a user process on the host OS.

    管理程序作为主机操作系统上的用户进程运行。

  • The virtual machine runs as a user process in user mode.

    虚拟机在用户模式下作为用户进程运行。

    • Is not allowed to execute “privileged instructions”.

      不允许执行“特权指令”。

  • The guest OS thinks it is running in kernel mode, although it is in user mode. (virtual kernel mode)

    来宾操作系统认为它在内核模式下运行，尽管它是在用户模式下。 （虚拟内核模式）

    • When the guest OS executes a “privileged instruction”, the hypervisor emulates it.

      当来宾操作系统执行“特权指令”时，虚拟机管理程序会模拟它。

7.3.3 Type 1 vs. Type 2 hypervisors 比较

与所期望的不同，类型 1 并非始终都是赢家。

It turns our that trap-and-emulate approach creates a lot of overhead due to the handling of the traps (including context switches) and that the binary translation approach (combined with caching of translated blocks) can run faster.

由于处理陷阱（包括上下文切换context switches），并且二进制转换binary translation方法（结合转换块的缓存）可以运行得更快，因此我们的trap-and-emulate方法会产生大量开销。

For this reason, some type 1 hypervisors also perform binary translation for speeding up.

出于这个原因，一些类型 1 管理程序也执行二进制转换以加快速度。

8. Process virtualization 进程虚拟化

Non-full system virtualization

Not all virtualization technology tries to trick the guest into believing that it has the entire system.

并非所有虚拟化技术都试图诱使客户相信它拥有整个系统。

Process-Level Virtualization allows a process to run that was originally written for a different operating system and/or architecture.

进程级虚拟化允许运行最初为不同操作系统和/或体系结构编写的进程。

Well-known examples:

  • WINE

    • compatibility layer that allow Windows applications to run on Linux.

      允许 Windows 应用程序在 Linux 上运行的兼容层

  • QEMU

    • emulator that allows applications for one architecture to run on another

      允许一种架构的应用程序在另一种架构上运行的模拟器

9. x86

Virtualization has long been a problem on the x86 architecture due to defects in the Intel 386 architecture that were slavishly carried forward into new CPUs for 20 years in the name of backward compatibility.

由于 Intel 386 架构中的缺陷，虚拟化长期以来一直是 x86 架构上的一个问题，20 年来，这些缺陷以向后兼容性的名义在新 CPU 中被盲目地推进。

On x86, some instructions are sensitive but not privileged.

在 x86 上，一些指令是敏感的但没有特权。

机器指令详细见 5.0.3 Machine instructions

9.0 Example: POPF (on Intel 386)

  • POPF replaces the flags register, which changes the bit that enables/disables interrupt. POPF 替换了标志寄存器，它改变了启用/禁用中断的位。

  • In user mode, this bit is simply not changed.

    在用户模式下，该位根本不会改变。

  • Is not privileged (does not trap).

    没有特权（不陷阱）。

  • Intel 386 and its successors could not be virtualized.

    Intel 386 及其后继产品无法虚拟化。

9.1 Techniques for virtualization on x86

具体见 4. Types

Solutions to Full virtualization of x86 CPU

  • Full description of operations of all x86 hardware (but they evolve)

    • A guest OS can run unchanged under the VMM as if it was running directly on the hardware platform. Each VM runs an exact copy of the actual hardware.

      来宾操作系统可以在 VMM 下原样运行，就像直接在硬件平台上运行一样。 每个 VM 都运行实际硬件的精确副本。

  • Binary translation (almost established)

    • rewrites parts of the code on the fly to replace sensitive but not privileged instructions with safe code to emulate the original instruction

      二进制翻译即时重写部分代码，用安全代码替换敏感但非特权指令以模拟原始指令

    • It is very difficult to build sophisticated binary translation support necessary for full virtualization.

      构建完全虚拟化所需的复杂二进制转换支持非常困难。

  • OS-assisted or paravirtualization

    • Paravirtualization is to modify the source code of the guest OS such that it makes hypervisor calls instead of executing “privileged instructions”.

      半虚拟化是修改来宾操作系统的源代码，使其调用管理程序hypervisor而不是执行“特权指令”。

  • Hardware-assisted (future direction)

    • Hardware switch supported by CPU.

      CPU 支持的硬件切换

    • Privileged and sensitive calls are set to automatically trap to the hypervisor, removing the need for either binary translation or paravirtualization.

      特权和敏感调用被设置为自动捕获到管理程序，消除对二进制转换或半虚拟化的需要。

9.2 Summary

  • Binary translation is the most established technology for full virtualization

    二进制翻译是完全虚拟化最成熟的技术

  • Paravirtualization delivers performance benefits with maintenance costs

    半虚拟化以维护成本提供性能优势

  • Hardware assist is the future of virtualization, but it still has a long way to go

    硬件辅助是虚拟化的未来，但仍有很长的路要走

10.3 VMM virtualizes the CPU and the memory

Hypervisors/VMM:

  • Traps the privileged instructions executed by a guest OS and enforces the correctness and safety of the operation

    捕获由来宾操作系统执行的特权指令并强制执行操作的正确性和安全性

  • Traps interrupts and dispatches them to the individual guest operating systems

    捕获中断并将它们分派到各个客户操作系统

  • Controls the virtual memory management

    控制虚拟内存管理

  • Maintains a shadow page table for each guest OS and replicates any modification made by the guest OS in its own shadow page table.

    This shadow page table points to the actual page frame and it is used by the Memory Management Unit (MMU) for dynamic address translation.

    为每个来宾 OS 维护一个影子页表，并在其自己的影子页表中复制来宾 OS 所做的任何修改。

    这个影子页表指向实际的页框，它被内存管理单元 (MMU) 用于动态地址转换。

  • Monitors the system performance and takes corrective actions to avoid performance degradation. For example, the VMM may swap out a VM to avoid thrashing

    监控系统性能并采取纠正措施以避免性能下降。 例如，VMM 可能会换出 VM 以避免颠簸
