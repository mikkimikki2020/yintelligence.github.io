Embedded operating systems 2:11:36

On embedded systems, an O.S. is an optional software layer providing low-level services to a program (application).

  • Platform-specific hardware abstraction layers
  • File management, disk access
  • Keyboard/display interfacing
  • Integrated power-management systems
  • Device drivers for common peripherals
  • Scheduling multiple programs for execution
    • Or even just multiple threads from one program
  • Program makes system calls to the OS

Types (Skipped in video)

The main types of embedded OS are as follows:

  • Single System Control Loop: Such a operating system is designed to run the only single task.
  • Multi-Tasking Operating System: As the name suggests that this operating system can perform multiple tasks.
  • Rate Monotonic Operating System: It is a type of operating system that ensures that task runs in a system can run for a specific interval of time and for a specific period of time.
  • Preemptive Operating System: A preemptive operating system is a type of multitasking operating system that interprets the preemptive predominance for tasks. A higher priority is task is always defined and run before a lower priority task.
  • Real Time Operating System: A real-time operating system is the one which serves real time applications.

Categories of embedded devices 2:13:01

Class 1: These devices are able to talk to each other but via constrained protocols.

Class 2: These devices are the less constrained and can use the same stacks of protocols used in personal computers and servers.

3. What is a Real-Time System? 2:16:54

‚ÄúReal-time programs or systems must guarantee response with a specified time constraint (aka deadlines) imposed by environment‚Äù

A system is said to be Real Time if it is required to complete it‚Äôs work & deliver it‚Äôs services on time.

Result in severe consequences if logical and timing correctness are not met

Example ‚Äì Flight Control System

  • All tasks in that system must execute on time.

Non Example ‚Äì PC system

Embedded system: is a computer system that performs a limited set of specific functions. It often interacts with its environment.

RTS: Correctness of the system depends not only on the logical results, but also on the time in which the results are produced.

Control systems

Example 1 2:19:32

A simple one-sensor, one-actuator control system

Pseudo-code for this system:

set timer to interrupt periodically with period T;
at each timer interrupt do
    do analog-to-digital conversion to get y;
    compute control output u;
    output u and do digital-to-analog conversion;
end do

T is called the sampling period. T is a key design choice.

Typical range for T: seconds to milliseconds.

    • TV video rate is 30 frames/sec. HDTV is 60 frames/sec.
    • Telephone audio is 16 Kbits/sec. CD audio is 128 Kbits/sec.
  • Other requirements: Lip synchronization, low jitter, low end-to-end response times (if interactive).
  • General examples:
    • Video capture.
    • Digital filtering.
    • Video and voice compression/decompression.
    • Radar signal processing.
  • Response times range from a few milliseconds to a few seconds.
  • Typically simpler than control systems

Real-Time System

Types

Soft real-time systems 2:27:40
  • Tasks are performed as fast as possible
  • Late completion of jobs (with some acceptably low probability) is undesirable but not fatal.
  • System performance degrades as more & more jobs miss deadlines
  • Also known as ‚Äúbest effort‚Äù systems
  • Example:
    • web sites and services
    • computer games
Firm real-time systems 2:28:23
  • Tasks are performed as fast as possible
  • Late completion of jobs is undesirable but not fatal.
  • Results discarded after deadline.
Hard real-time systems 2:29:16
  • Tasks have to be performed on time
  • Failure to meet deadlines is fatal
  • Example:
    • Flight Control System
    • Vehicle subsystems control

Definitions

Worst case execution time

Definition: The Worst Case Execution Time (WCET) is an upper bound on the execution time of a task.

  • Computing such a bound is undecidable
    • Possible for programs without recursion and finite loops
      • Pipeline hazards, interrupts, caches -> serious overestimates

Approaches:

  • for hardware: typically requires hardware synthesis
  • for software: requires availability of machine programs; complex analysis
Average execution time

Estimated cost and performance values

  • Difficult to generate sufficiently precise estimates; Balance between run-time and precision

Accurate cost and performance values

  • Can be done with normal tools (such as compilers). As precise as the input data is.

Task periodicity

Period and deadline

Period of a task (if periodic)

  • Repeating time interval the task must complete one execution within
    • E.g., period = 100 ms
    • Task must execute once every 100 ms
  • Usually determined by the description of the system
    • E.g., refresh rate of display is 27 times/sec
    • Period = 37 ms

Execution deadline

  • Amount of time task must be completed by after it has started
    • E.g., execution time = 5 ms, deadline = 20 ms, period = 100 ms
    • Task must complete execution within 20 ms after it has begun regardless of its period
    • Task begins at start of period, runs for 4 ms then is preempted
    • Task suspended for 14 ms, then runs for the remaining 1 ms
    • Completed within 4 + 14 + 1 = 19 ms which meets deadline of 20 ms
    • Without deadline process could be suspended for much longer

Periodic Task repeats itself regularly after a certain fixed time interval.

Periodic tasks are sometimes referred to as clock-driven tasks

Task scheduling

Scheduler

Real-time systems must meet timing requirements when multiple concurrent processes implemented on single general-purpose processor

  • Not true multitasking

Scheduler is a special process that decides when and for how long each process is executed

The main role of task scheduling refers to identifying the order in which tasks should execute in a system.

A schedule for a given set of tasks is thus an assignment of time-frames and resources to individual tasks.

There exist multiple scheduling algorithms.

  • The time instants at which scheduling decisions are taken are called scheduling points.
  • Their quality is identified by a term called processor utilization.

A good scheduling algorithm should lead to 100% processor utilization.

Classification

Based up to scheduling points, scheduling algorithms can be classified into the following categories:

  • Clock driven scheduling: the scheduling points are the interrupts received from a periodic clock
    • Table driven
    • Cyclic
  • Event driven scheduling: the scheduling points correspond to different events in the system such as when an instance of a task finishes execution.
    • Simply priority scheduling
    • Rate monotonic scheduling (RMS)
    • Earliest deadline first (EDF) scheduling
Clock driven scheduling

This type of schedulers are called static schedulers.

Clock driven scheduler works in synchronism with clock or timer signal, which periodically generates interrupts.

Simplified scheduled due to knowing in advance:

  • The set of tasks
  • Their periodicity values
  • Execution times required
  • Deadlines

Drawback: this type of schedulers is their inability to handle aperiodic and sporadic tasks

Types:

  • Table driven
  • Cyclic
Table driven scheduling

Table driven schedulers use a precomputed table that stores the tasks to be run at different clock intervals.

Cyclic scheduling

A major problem with table-driven scheduling is the size of the schedule table.

In cyclic scheduling, the major cycle (equal to the least common multiple of periodicity tasks) is divided into a number of equal sized minor cycles or frames.

One or more frames are allocated to individual tasks.

The schedule table stores the tasks to be run for different frames, so the size of the tables is reduced to be equal to the number of frames.

The selection of the frame size becomes an issue:

  • A large frame size implies lesser number of frames in a major cycle.
  • It may lead to wastage of CPU time as for most of the tasks, execution time may be much smaller than the frame size.

The constraints behind selection of a frame size are the following:

  • Minimize context switching
  • Schedule table minimization
  • Satisfaction of task deadlines
Limitations

Problems of clock driven scheduling strategies:

  • their inability to handle a large number of tasks.
  • determining the proper frame size
  • sporadic and aperiodic tasks cannot be handled efficiently
Priorities

Tasks with highest priority always selected first by scheduler

  • Typically determined statically during creation or dynamically during execution

Priority queue

  • Runnable tasks again added as created or become runnable
  • Task with highest priority chosen when new task needed
  • If multiple tasks with same highest priority value then selects from them using first-come first-served
Event driven scheduling

The scheduling of sporadic and aperiodic tasks have led to the design of event driven schedulers

Priorities are assigned to tasks.

There are three important schedulers in this category:

  1. Foreground-background scheduling
  2. Rate Monotonic scheduling (RMS)
  3. Earliest Deadline First (EDF) scheduling
Foreground-background scheduling

Small systems of low complexity

These systems are also called ‚Äúsuperloops‚Äù

An application consists of an infinite loop of desired operations (background)

Interrupt service routines (ISRs) handle asynchronous events (foreground)

Critical operations must be performed by the ISRs to ensure the timing correctness

Thus, ISRs tend to take longer than they should

Rate Monotonic scheduling

Rate Monotonic Scheduling is a static-priority-based event driven scheduling algorithm for periodic tasks.

The priorities are assigned to the tasks based on their periodicity values.

Rate monotonic scheduling

  • Processes with shorter periods have higher priority
  • Typically used when execution deadline = period

The theory underlying Rate Monotonic Scheduling is the Rate Monotonic Analysis, that uses simple model of the system stated as follows:

  1. All Processes run on a single CPU, thus there is no task parallelism
  2. Context switching time ignored
  3. Execution time for different invocations of a task are same and constant
  4. Tasks are totally independent of each other
  5. The deadline of an instance of a task occurs at the end of its period

Advantages:

  • The major advantage of Rate Monotonic Scheduling is its simplicity.
  • Since the priorities are static, a simple array of tasks can be used as the data structure.
  • Another good feature is its ability to handle transient overloads efficiently. It cannot lead to higher priority tasks missing deadlines.
  • Almost all the commercial real-time operating systems support Rate Monotonic Scheduling policy

Disadvantages:

  • It is difficult to handle aperiodic and sporadic tasks.
  • Rate Monotonic Scheduling does not produce optimal results if deadlines differ from periodicity of tasks.
  • In Deadline Monotonic Algorithm priorities are assigned to the tasks based on their deadlines
    • Processes with shorter deadlines have higher priority
    • Typically used when execution deadline < period
    • If deadlines are different from periodicity, there may be cases in which Rate Monotonic Scheduling fails but Deadline Monotonic Scheduling produce valid schedules.
    • On the contrary, wherever Deadline Monotonic Scheduling fails, Rate Monotonic Scheduling also fails
Deadline Monotonic scheduling

Deadline Monotonic Scheduling is a static-priority-based event driven scheduling algorithm for periodic tasks that assigns priorities to the tasks based on their deadlines values.

Deadline monotonic scheduling

  • Processes with shorter deadlines have higher priority
  • Typically used when execution deadline < period

Earliest Deadline Monotonic scheduling

Earliest Deadline First Scheduling is a dynamic priority algorithms in which priority of an instance of a task depends on its deadline.

Unlike Rate Monotonic scheduling, the priority of a task is not fixed for all instances.

The priority changes dynamically during the lifetime of the system.

Disadvantages:

  • Earliest Deadline First Scheduling is less predictable.
  • Earliest Deadline First Scheduling is less controllable in the sense that we cannot control the response time of a task
  • Implementation overhead is higher than fixed priority approaches.
  • Transient overload problem
  • Domino effect

3.1 Role of an OS in Real-Time Systems

RTOS provide mechanisms, primitives, and guidelines for building real-time systems

Standalone Applications

  • Often no OS involved(bare metal)
  • Micro-controller based Embedded Systems

But many Real Time Applications are huge & complex

  • Multiple threads
  • Complicated Synchronization Requirements
  • File system / Network / Windowing support
  • OS primitives reduce the software design time

Real-time operating systems are specialised operating systems which manage the processes in the RTS.

Responsible not only for process management and resource (processor and memory) allocation.

Do not normally include facilities such as file management.

3.2 Why use an RTOS?

Here are important reasons for using RTOS:

  • It offers priority-based scheduling, which allows you to separate analytical processing from non-critical processing.
  • The Real time OS provides API functions that allow cleaner and smaller application code.
  • Abstracting timing dependencies and the task-based design results in fewer interdependencies between modules.
  • RTOS offers modular task-based development, which allows modular task-based testing.
  • The task-based API encourages modular development as a task, will typically have a clearly defined role. It allows designers/teams to work independently on their parts of the project.
  • An RTOS is event-driven with no time wastage on processing time for the event which is not occur

3.4 Advantages of RTOS

  • Maximum consumption: Maximum utilization of devices and system. Thus more output from all the resources.
  • Task Shifting: Time assigned for shifting tasks in these systems are very low.
  • Focus On Application: Focus on running applications and less importance to applications which are in queue.
  • Real Time Operating System In Embedded System: Since size of programs are small, RTOS can also be embedded systems like in transport and others.
  • Error Free: These types of systems are error free.
  • Memory Allocation: Memory allocation is best managed in these type of systems

  • Memory management
  • Interrupt service mechanism
  • I/O management

  • Communication subsystems (Option)
  • Board Support Packages (BSP)

3.6 Components of RTOS

  • The Scheduler
  • Symmetric Multiprocessing (SMP)
  • Function Library
  • Memory Management
  • Fast dispatch latency
  • User-defined data objects and classes

3.7 Popular Real-Time Operating Systems

  • FreeRTOS
    • Scalable real time kernel designed specifically for small embedded systems.
    • RTOS Scheduler - preemptive, cooperative and hybrid configuration options, with optional time slicing.
    • Round robin with time slicing, mutexes with priority inheritance and recursive mutexes.
    • 6K to 12K ROM footprint
  • Neutrino/QNX
    • Real-time microkernel surrounded by optional processes (resource managers) that provide POSIX and UNIX compatibility
    • Microkernels typically support only the most basic services
    • Optional resource managers allow scalability from small ROM-based systems to huge multiprocessor systems connected by various networking and communication technologies
    • Preemptive process scheduling using FIFO, round-robin, adaptive, or priority-driven scheduling
    • 32 priority levels per process
    • Microkernel < 10 Kbytes and complies with POSIX real-time standard

3.8 Factors for selecting an RTOS

Performance: Performance is the most important factor required to be considered while selecting for a RTOS.

Middleware: if there is no middleware support in Real time operating system, then the issue of time-taken integration of processes occurs.

Error-free: RTOS systems are error-free. Therefore, there is no chance of getting an error while performing the task.

Embedded system usage: Programs of RTOS are of small size. So we widely use RTOS for embedded systems.

Maximum Consumption: we can achieve maximum consumption with the help of RTOS.

Task shifting: Shifting time of the tasks is very less.

Unique features: A good RTS should be capable, and it has some extra features like how it operates to execute a command, efficient protection of the memory of the system, etc.

3.9 What's important in Real-Time

Metrics for real-time systems differ from that for time-sharing systems

  • Schedulability is the ability of tasks to meet all hard deadlines
  • Latency is the worst-case system response time to event
  • Stability in overload means the system meets critical deadlines even if all deadlines cannot be met

3.10 Disadvantages of RTOS

Limited tasks: RTOS system can run minimal tasks together, and it concentrates only on those applications which contain an error so that it can avoid them.

Poor multi-tasking: RTOS is the system that concentrates on a few tasks. Therefore, it is really hard for these systems to do multi-tasking.

Limited switching: Minimum switching of tasks is done in Real time operating systems.

Device Driver And Interrupt signals: Specific drivers are required for the RTOS so that it can offer fast response time to interrupt signals, which helps to maintain its speed.

Use Heavy System Resources: Plenty of resources are used by RTOS, which makes this system expensive.

Thread Priority: The tasks which have a low priority need to wait for a long time as the RTOS maintains the accuracy of the program, which are under execution.

Complex Algorithm: The algorithms are very complex and difficult for the designer

4. Summary

  • RTOS is an operating system intended to serve real time application that process data as it comes in, mostly without buffer delay.
  • It offers priority-based scheduling, which allows you to separate analytical processing from noncritical processing.
  • Important components of RTOS system are:
    1. The Scheduler
    2. Symmetric Multiprocessing
    3. Function Library
    4. Memory Management
    5. Fast dispatch latency
    6. User-defined data objects and classes
  • Three types of RTOS:
    1. Hard time
    2. Soft time
    3. Firm time
  • RTOS system occupy very less memory and consume fewer resources
  • Performance is the most important factor required to be considered while selecting for a RTOS.
  • General-Purpose Operating System (GPOS) is used for desktop PC and laptop while Real-Time Operating System (RTOS) only applied to the embedded application.
  • Real-time systems are used in Airlines reservation system, Air traffic control system, etc.
  • A major drawback of RTOS is that the system only concentrates on a few tasks.
