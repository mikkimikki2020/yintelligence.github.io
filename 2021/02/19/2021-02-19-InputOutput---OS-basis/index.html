1. Principles of I/O hardware 59:33

1.1 IO devices

Different people look at I/O hardware in different ways:

  • Electrical engineers look at it in terms of chips, wires, power supplies, motors, and all the other physical components that comprise the hardware.
  • Programmers look at the interface presented to the software.

I/O devices can be divided into two categories:

  • block devices
  • character devices.

This classification scheme is not perfect because some devices do not fit in, but the model of block and character devices is general enough that it can be used as a basis.

1.2 Device controllers 1:02:14

I/O devices have components:

  • mechanical component (the device itself)
  • electronic component (device controller)

The electronic component is the device controller

  • may be able to handle multiple devices

Controller's tasks:

  • convert serial/parallel bit stream to block of bytes
  • perform error correction as necessary
  • make available to main memory

I/O Communication:

  • Memory-Mapped I/O
  • Direct Memory Accesses
  • Interruptions

2. Memory-mapped I/O 内存映射 1:03:41

How the CPU communicates with the control registers and also with the device data buffers?

  1. Separate I/O and memory space (Standard I/O or I/O-mapped I/O)
  2. Memory-mapped I/O
  3. Hybrid

2.1 x86

  1. A single-bus architecture (old architectures)
  2. A dual-bus memory architecture (current architectures)

3. Direct Memory Access 直接内存访问 (DMA) 1:09:45

3.1 DMA Features 1:14:25

DMA controllers vary considerably in their sophistication.

More complex DMA controllers can be programmed to handle multiple transfers at the same time.

Many buses can operate in two modes:

  • word-at-a-time (former) mode => cycle stealing
  • block mode => burst mode

Not all computers (or embedded systems) use DMA.

4. Interrupt-driven I/O using vectored interrupt 1:16:49

4.1 Without interrupt controller

When an interruption happens, the processor suspends current program, jumps to an Interrupt Service Routine, or ISR

What is the address (interrupt address vector) of the ISR?

Vectored interrupt

  • Peripheral must provide the address
  • Common when microprocessor has multiple peripherals connected by a system bus

4.2 With an interrupt controller 1:21:20

Connections between devices and interrupt controller actually use interrupt lines on the bus rather than dedicated wires

5. Principles of I/O hardware 1:22:46

5.1 Goals of I/O software

Device independence

  • programs can access any I/O device
  • without specifying device in advance
    • (floppy, hard drive, or CD-ROM)

Uniform naming

  • name of a file or device a string or an integer
  • not depending on which machine

Error handling

  • handle as close to the hardware as possible

Synchronous vs. asynchronous transfers.

  • Asynchronous (interrupt-driven): CPU starts, but continues processing until the interruption occurs
    • Easier
    • High-performance applications
  • Synchronous (blocking)

Buffering

  • Decouple data rates

Sharable vs. dedicated devices

There are three fundamentally different ways that I/O can be performed from the software perspective:

  • Programmed I/O (simplest)
  • Interrupt-driven I/O
  • I/O using DMA

5.1.1 Programmed I/O 1:26:26

5.1.2 Interrupt-driven I/O 1:28:43

Interrupts allow CPU to do something else while waiting for the I/O device.

Writing a string to the printer using interrupt-driven I/O.

  1. Code executed at the time the print system call is made.
  2. Interrupt service procedure for the printer

5.1.3 I/O using DMA 1:29:32

An obvious disadvantage of interrupt-driven I/O is that an interrupt occurs on every character.

Interrupts take time, so this scheme wastes a certain amount of CPU time.

DMA controller feed the characters to the printer one at time, without the CPU being bothered.

Printing a string using DMA.

  1. Code executed when the print system call is made.
  2. Interrupt-service procedure.

6. I/O software layers 1:30:38

I/O software is typically organized in four layers

  1. Interrupt Handlers
  2. Device drivers
  3. Device-independent operating system software
  4. User-level I/O software

6.1 Interrupt handlers 1:31:16

Interrupt handlers are best hidden

  • have driver starting an I/O operation block until interrupt notifies of completion

Interrupt procedure does its task

  • then unblocks driver that started it

Steps must be performed in software after interrupt completed

6.2 Device drivers 1:33:18

Logical position of device drivers is shown here

Communications between drivers and device controllers goes over the bus

  • Operating systems define a standard interface
  • Different systems handle loading drivers in different ways.
  • A device driver has several functions.
    • Communicate with the device-independent software
    • Initialize the device
    • Manage its power requirements
    • Log events
  • Device driver functionality
    1. A typical driver starts out by checking the input parameters to see if they are valid.
    2. The driver may check if the device is currently in use.
    3. Controlling the device means issuing a sequence of commands to it.
    4. After the commands have been issued, one of two situations will apply.
      1. the device driver must wait until the controller does some work for it
      2. the operation finishes without delay, so the driver need not block
    5. After the operation has been completed, the driver must check for errors
    6. It returns some status information for error reporting back to its caller.
    7. Finally, driver executes the request or waits for the next one

6.3 Device-Independent I/O software 1:37:58

Although some of the I/O software is device specific, other parts of it are device independent.

The basic function of the device-independent software is to perform the I/O functions that are common to all devices and to provide a uniform interface to the user-level software.

Functions of the device-independent I/O software

6.3.1 Uniform interfacing 1:38:41

A major issue in an operating system is how to make all I/O devices and drivers look similar.

One aspect of this issue is the interface between the device drivers and the rest of the operating system.

  1. Without a standard driver interface
  2. With a standard driver interface

Uniform interfacing works as follows:

  • Functions: For each class of devices the operating system defines a set of functions that the driver must supply.
  • Naming: The device-independent software takes care of mapping symbolic device names onto the proper driver.
    • For example, in UNIX a device name, such as /dev/disk0
  • Protection: The system administrator can then set the proper permissions for each device.
