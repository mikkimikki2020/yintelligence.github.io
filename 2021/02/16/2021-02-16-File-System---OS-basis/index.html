Long-term information storage

  • Must store large amounts of data
  • Information stored must survive the termination of the process using it
  • Multiple processes must be able to access the information concurrently

Files 3:01

Just a few of the questions that quickly arise are:

  1. How do you find information?
  2. How do you keep one user from reading another user’s data?
  3. How do you know which blocks are free?

and there are many more.

Files are logical units of information created by processes.

A disk will usually contain thousands or even millions of them, each one independent of the others.

Information stored in files must be persistent, that is, not be affected by process creation and termination.

Files are managed by the operating system. How they are structured, named, accessed, used, protected, implemented, and managed are major topics in operating system design.

Disk space management 33:05

Files are normally stored on disk, so management of disk space is a major concern to filesystem designers.

Strategies:

  1. n consecutive bytes of disk space are allocated
  2. the file is split up into a number of (not necessarily) contiguous blocks

The same trade-off is present in memory-management systems between pure segmentation and paging.

Problem: storing a file as a contiguous sequence of bytes has the obvious problem that if a file grows, it may have to be moved on the disk.

For this reason, nearly all file systems chop files up into fixed-size blocks that need not be adjacent

Size block 34:15

What is the most adequate block size?

If the allocation unit is too large, we waste space; if it is too small, we waste time.

  • Solid line (right hand scale) shows the space efficiency as a function of block size.
  • Dashed line (left hand scale) shows the data rate for such a disk as a function of block size.
  • For simplicity, let us assume that all files are 4 KB.

What the curves show, however, is that performance and space utilization are inherently in conflict.

Small blocks are bad for performance but good for diskspace utilization.

File system reliability 42:17

Backups

  • Destruction of a file system is often a far greater disaster than destruction of a computer.
  • Largely sub-estimated.
  • Backups to tape are generally made to handle one of two potential problems:
    1. Recover from disaster.
    2. Recover from stupidity.
  • Incremental backups or dumps
  • Two strategies can be used for dumping a disk to a backup disk:
    • a physical dump
      • Unused blocks
      • Bad blocks
    • a logical dump.

File system performance 50:27

Access to disk is much slower than access to memory.

Three techniques to improve performance.

  • Caching
  • Block read Ahead
  • Reducing Disk-Arm Motion

Caching 50:47

A cache is a collection of blocks that logically belong on the disk but are being kept in memory for performance reasons.

The block cache data structures:

LRU: Least Recent Used

MRU: Most Recent Used

Block read ahead 52:40

A second technique for improving perceived file-system performance is to try to get blocks into the cache before they are needed to increase the hit rate.

  • This read-ahead strategy works only for files that are actually being read sequentially
  • If a file is being randomly accessed, read ahead does not help
    • disk bandwidth reading in useless blocks
    • removing potentially useful blocks from the cache
  • Continuous monitoring of the file access pattern

Another important technique 53:45

Another important technique is to reduce the amount of disk-arm motion by putting blocks that are likely to be accessed in sequence close to each other, preferably in the same cylinder.

  1. I-nodes placed at the start of the disk
  2. Disk divided into cylinder groups
    • each with its own blocks and i-nodes
