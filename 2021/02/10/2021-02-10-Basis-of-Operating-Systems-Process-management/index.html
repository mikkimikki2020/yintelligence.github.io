Basis of OS - Process management

Video 50:30 ~ 2:11:37

Processes

What’s in a Process?

A process consists of:

  • Address Space
  • One or more threads of control executing in that address space
  • Additional system state associated with it
    • Open files
    • Open sockets (network connections)
    • …

The process model

A (sequential) process = an instance of an executing program, including the current values of the program counter, registers and variables.

A sequential program, typically an infinite loop

  • Executes concurrently with other processes

  • We are about to enter the world of “concurrent programming”

Basic operations on processes 58:23

  • Create and terminate
    • Create is like a procedure call but caller doesn’t wait
    • Terminate kills a process, destroying all data
  • Suspend and resume
    • Suspend puts a process on hold, saving state for later execution
    • Resume starts the process again where it left off
  • Join
    • A process suspends until a particular child process finishes execution

Process creation and termination

Events which cause process creation:
  • System initialization: launches foreground processes (interacting with users) and background processes (typically called daemons)
  • Execution of a process creation system call by a running process.
  • A user request to create a new process.
  • Initiation of a batch job.
Events which cause process termination:
  • Normal exit (voluntary).
  • Error exit (voluntary).
  • Fatal error (involuntary).
  • Killed by another process (involuntary).

Process states 1:05:36

  • Running state: actually using the CPU at that instant

  • Ready state: runnable; temporarily stopped to let another process run

  • Blocked state: unable to run until some external event happens

Implementation of processes 1: 09:24

The lowest layer of a process-structured operating system handles interrupts and scheduling.

Above that layer are sequential processes.

The operating system maintains a table, called the process table.

Contains one entry per process.

Skeleton of what the lowest level of the operating system does when an interrupt occurs.

Threads 1:12:36

Thread = light-weight processes, created by a mother process (program) for pseudo-parallel processing.

No memory protection: share memory with each other and the mother process.

Thread usage

The classical thread model 1:18:33

The first column lists some items shared by all threads in a process.

The second one lists some items private to each thread.

Each thread has its own stack.

Posix Threads (How to make a thread? 1:24:42)

IEEE standard 1003.1c: enables writing of portable threaded programs

The threads package it defines is called Pthreads

The standard defines over 60 function calls, which is far too many to go over here.

Implementing threads in user space or in kernel 1:29:17

  1. A user-level threads package.

  2. A threads package managed by the kernel.

Processes vs Threads

Regular processes

  • Heavyweight process
  • Own virtual address space (stack, data, code)
  • System resources (e.g., open files)

Threads

  • Lightweight process
  • Sub-process within process
  • Only program counter, stack, and registers
  • Shares address space, system resources with other threads
    • Allows quicker communication between threads
  • Small compared to heavyweight processes
    • Can be created quickly
    • Low cost switching between threads

Interprocess communication (IPC) 1:36:36

Race conditions

Two processes want to access shared memory at the same time.

Race condition if the CPU decides to switch from the just initiated Process A, that has just received the information that in=7, to Process B that will also observe in=7. Both will write their file at the same location in the printer spooler directory.

Critical regions

Conditions required to avoid race condition:

  • No two processes may be simultaneously inside their critical regions.
  • No assumptions may be made about speeds or the number of CPUs.
  • No process running outside its critical region may block other processes.
  • No process should have to wait forever to enter its critical region.

Mutual exclusion with busy waiting

Proposals for achieving mutual exclusion:

  • Disabling interrupts
    • Problems: race conditions, too much control to the user, multicore systems (interrupts handled per core)
  • Lock shared variables
    • lock=0: no process in critical region; lock=1: process in critical region
    • Problem: race conditions remain (cfr. spooler directory)

Several alternatives:

  • Strict alternation
  • Peterson's solution
  • The Test and Set Lock (TSL) instruction

Communication among processes

  • Processes need to communicate data and signals to solve their computation problem.

    ​ -> Processes that don’t communicate are just independent programs solving separate problems

  • Basic example: producer/consumer

    • Process A produces data items, Process B consumes them
    • E.g., A decodes video packets, B display decoded packets on a screen

The producer-consumer problem (or bounded-buffer problem) with a fatal race condition -> Possible solution = wake-up waiting bits

How do we achieve this communication?

Shared memory: Processes read and write shared variables
  • No time overhead, easy to implement
  • But, hard to use – mistakes are common

Example: Producer/consumer with a mistake 1:46:14 -1:47:47

  • Share buffer[N], count

    ▶ count = # of valid data items in buffer

  • processA produces data items and stores in buffer

    ▶ If buffer is full, must wait

  • processB consumes data items from buffer

    ▶ If buffer is empty, must wait

  • Error when both processes try to update count concurrently (lines 10 and 19) and the following execution sequence occurs. Say “count” is 3.

    1. A loads count (count = 3) from memory into register R1 (R1 = 3)
    2. A increments R1 (R1 = 4)
    3. B loads count (count = 3) from memory into register R2 (R2 = 3)
    4. B decrements R2 (R2 = 2)
    5. A stores R1 back to count in memory (count = 4)
    6. B stores R2 back to count in memory (count = 2)

  • count now has incorrect value of 2

    data_type buffer[N];
int count = 0;
void processA() {
  int i;
  while( 1 ) {
    produce(&data);
    while( count == N );/*loop*/
    buffer[i] = data;
    i = (i + 1) % N;
    count = count + 1;
    }
 }
void processB() {
  int i;
  while( 1 ) {
    while( count == 0 );/*loop*/
    data = buffer[i];
    i = (i + 1) % N;
    count = count - 1;
    consume(&data);
   }
}
void main() {
  create_process(processA);
  create_process(processB);
}

Safer model, but less flexible

  • Sending process performs special operation, send

  • Receiving process must perform special operation, receive, to receive the data

  • Both operations must explicitly specify which process it is sending to or receiving from

  • Receive is blocking, send may or may not be blocking

Shared memory and mutexes
Mutual exclusion

  • Certain sections of code should not be performed concurrently

    Critical section:

    ​ Possibly noncontiguous section of code where simultaneous updates, by multiple processes to a shared memory location, can occur

  • When a process enters the critical section, all other processes must be locked out until it leaves the critical section

    Mutex

    • A shared object used for locking and unlocking segment of shared data
    • Disallows read/write access to memory it guards
    • Multiple processes can perform lock operation simultaneously, but only one process will acquire lock
    • All other processes trying to obtain lock will be put in blocked state until unlock operation performed by acquiring process when it exits critical section
    • These processes will then be placed in runnable state and will compete for lock again
Correct solution 1:51:32 -1:53:29

The primitive mutex is used to ensure critical sections are executed in mutual exclusion of each other

Following the same execution sequence as before:

  1. A/B execute lock operation on count_mutex
  2. Either A or B will acquire lock
    1. Say B acquires it
    2. A will be put in blocked state
  3. B loads count (count = 3) from memory into register R2 (R2 = 3)
  4. B decrements R2 (R2 = 2)
  5. B stores R2 back to count in memory (count = 2)
  6. B executes unlock operation
    1. A is placed in runnable state again
  7. A loads count (count = 2) from memory into register R1 (R1 = 2)
  8. A increments R1 (R1 = 3)
  9. A stores R1 back to count in memory (count = 3)
  10. Count now has correct value of 3
data_type buffer[N];
int count = 0;
mutex count_mutex;
void processA() {
  int i;
  while( 1 ) {
    produce(&data);
      while( count == N );/*loop*/
      buffer[i] = data;
      i = (i + 1) % N;
      count_mutex.lock();
      count = count + 1;
      count_mutex.unlock();
    }
}
void processB() {
  int i;
  while( 1 ) {
    while( count == 0 );/*loop*/
    data = buffer[i];
    i = (i + 1) % N;
    count_mutex.lock();
    count = count - 1;
    count_mutex.unlock();
    consume(&data);
   }
}
void main() {
  create_process(processA);
  create_process(processB);
}
Semaphore

A semaphore is a variable or abstract data type that provides a simple but useful abstraction for controlling access by multiple processes to a common resource in a parallel programming environment.

Mutex VS Semaphore

  • A Mutex is different than a semaphore as it is a locking mechanism while a semaphore is a signaling mechanism.

    wait (mutex);

Critical section

  • A binary semaphore can be used as a Mutex but a Mutex can never be used as a semaphore.

  • • A semaphore is a signalling mechanism and a thread that is waiting on a semaphore can be signaled by another thread. • This is different than a mutex as the mutex can be signaled only by the thread that called the wait function.

Synchronization among processes

Sometimes concurrently running processes must synchronize their execution:

​ When a process must wait for:

  • another process to compute some value
  • reach a known point in their execution
  • signal some condition

Recall producer-consumer problem:

  • processA must wait if buffer is full
  • processB must wait if buffer is empty
  • This is called busy-waiting
    • Process executing loops instead of being blocked
    • CPU time wasted

More efficient methods:

  • Join operation, and blocking send and receive discussed earlier
    • Both block the process so it doesn’t waste CPU time
  • Condition variables and monitors

Condition variables 1:56:50

Condition variables are a synchronization mechanism available in Pthreads.

A condition variable is an object that has 2 operations, signal and wait.

When process performs a wait on a condition variable, the process is blocked until another process performs a signal on the same condition variable.

How is this done?

  • Process A acquires lock on a mutex
  • Process A performs wait, passing this mutex
    • Causes mutex to be unlocked
  • Process B can now acquire lock on same mutex
  • Process B enters critical section
    • Computes some value and/or make condition true
  • Process B performs signal when condition true
    • Causes process A to implicitly reacquire mutex lock
    • Process A becomes runnable

Example 1:58:56

2 condition variables:

  1. buffer_empty : Signals at least 1 free location available in buffer
  2. buffer_full : Signals at least 1 valid data item in buffer

processA:

  • produces data item

  • acquires lock (cs_mutex) for critical section

  • checks value of count

  • if count = N, buffer is full

    • performs wait operation on buffer_empty
    • this releases the lock on cs_mutex allowing processB to enter critical section, consume data item and free location in buffer
    • processB then performs signal

  • if count < N, buffer is not full

    • processA inserts data into buffer

    • increments count

    • signals processB making it runnable if it has performed a wait operation on buffer_full

using condition variables
data_type buffer[N];
int count = 0;
mutex cs_mutex;
condition buffer_empty, buffer_full;
void processA() {
  int i;
  while( 1 ) {
    produce(&data);
    cs_mutex.lock();
    if( count == N ) buffer_empty.wait(cs_mutex);
    buffer[i] = data;
    i = (i + 1) % N;
    count = count + 1;
    cs_mutex.unlock();
    buffer_full.signal();
   }
}
void processB() {
  int i;
  while( 1 ) {
    cs_mutex.lock();
    if( count == 0 ) buffer_full.wait(cs_mutex);
    data = buffer[i];
    i = (i + 1) % N;
    count = count - 1;
    cs_mutex.unlock();
    buffer_empty.signal();
    consume(&data);
   }
}
void main() {
  create_process(processA); create_process(processB);
}

Monitors 2:00:37

Definition: Collection of data and methods or subroutines that operate on data similar to an object-oriented paradigm

Monitor guarantees only 1 process can execute inside monitor at a time

  1. Process X executes while Process Y has to wait
  2. Process X performs wait on a condition
    • Process Y allowed to enter and execute
  3. Process Y signals condition Process X waiting on
    • Process Y blocked
    • Process X allowed to continue executing
  4. Process X finishes executing in monitor or waits on a condition again
    • Process Y made runnable again

Example 2:02:57-2:04:37

Single monitor encapsulates both processes along with buffer and count

One process will be allowed to begin executing firs

If process B allowed to execute first:

  1. Will execute until it finds count = 0
  2. Will perform wait on buffer_full condition variable
  3. processA now allowed to enter monitor and execute
  4. processA produces data item
  5. finds count < N so writes to buffer and increments count
  6. processA performs signal on buffer_full condition variable
  7. processA blocked
  8. processB reenters monitor and continues execution, consumes data, etc.
Monitor {
  data_type buffer[N];
  int count = 0;
  condition buffer_full, condition buffer_empty;
  void processA() {
    int i;
    while ( 1 ) {
    produce(&data);
    if( count == N ) buffer_empty.wait();
    buffer[i] = data;
    i = (i + 1) % N;
    count = count + 1;
    buffer_full.signal();
   }
}
void processB() {
  int i;
  while( 1 ) {
    if( count == 0 ) buffer_full.wait();
    data = buffer[i];
    i = (i + 1) % N;
    count = count - 1;
    buffer_empty.signal();
    consume(&data);
    buffer_full.signal();
   }
 }
} /* end monitor */
void main() {
  create_process(processA); create_process(processB);
}

Scheduling

Video 1:54:50 ~ 2:07:00

Process behaviour 1:56:32

  1. A CPU/Compute-bound process.

  2. An I/O-bound process.

Must meet timing requirements when multiple concurrent processes implemented on single general-purpose processor.

  • Not true multitasking

Scheduler 1:58:08

  • Special is a special process that decides when and for how long each process is executed.
  • The main role of processes scheduling refers to identifying the order in which processes should execute in a system.
  • A schedule for a given set of processes is thus an assignment of time-frames and resources to individual processes

A task scheduler can be implemented as preemptive or non-preemptive scheduler based on the tasks

  • Preemptive
    • Determines how long a process executes before preempting to allow another process to execute
    • Time quantum: predetermined amount of execution time preemptive scheduler allows each process (may be 10 to 100s of milliseconds long)
    • It allows that the execution of an executing processes may be stopped if a higher priority process requires service.
    • Determines which process will be next to run
  • Non-preemptive
    • Once a process has been scheduled for execution, it runs to completion or until it is blocked for some reason (e.g. waiting for I/O).
    • Only determines which process is next after current process finishes execution

Scheduling algorithms and their goals 2:00:41

Some goals of different scheduling algorithms under different circumstances.

Scheduling in batch systems 2:02:06

First-come first-served

Shortest job first

Shortest remaining time next

  1. Running four jobs in the original order

  2. Running them in shortest job first order.

Scheduling in interactive systems 2:03:12

Round-robin scheduling

  1. The list of runnable processes

  2. The list of runnable processes after B uses up its quantum

Priority scheduling 2:04:10

Process with highest priority always selected first by scheduler

  • Typically determined statically during creation and dynamically during execution

First-In-First-Out (FIFO)

  • Runnable processes added to end of FIFO as created or become runnable
  • Front process removed from FIFO when time quantum of current process is up or process is blocked

Priority queue

Runnable processes again added as created or become runnable

Process with highest priority chosen when new process needed

If multiple processes with same highest priority value then selects from them using first-come firstserved

  • Called priority scheduling when non-preemptive
  • Called round-robin when preemptive
