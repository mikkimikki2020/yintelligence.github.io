Pushdown Automata

Introduction

  • The class of regular languages can be recognised by Finite Automata (FAs)
  • The class of context-free languages can be recognised by Pushdown Automata (PDAs)

PDAS

  • have states and transitions, like FAs
  • have an additional memory device: a pushdown stack

Stack-augmented state transition diagrams

  • PDA’s can be visualised using stack-augmented state transition diagrams
  • These look like regular state transition diagrams, but the the transition label consists now of:
    • an input symbol (current symbol in the input string)
    • a stack symbol (current symbol on top of stack)
    • stack action (pop from and/or push on stack / do nothing)

Diagram example

  • transitions: read from string, check on stack / stack action
  • ◁ represents bottom of stack, - represents no stack change
  • Recognizes language L = anbn , I is used to count the As

Formal PDA notion

Introduction

2 changes as compared to stack-augmented state transition diagrams:

  1. Stack action:
    • Stack is always popped as an automatic first stage in the stack action of each transition
    • Any possible stack action is pushing a string of 0 or more symbols
  2. Transition function: I Input: 3 arguments: state, input symbol, stack symbol I Output: ordered pairs of next state and string of stack symbols to push
    • Input: 3 arguments: state, input symbol, stack symbol
    • Output: ordered pairs of next state and string of stack symbols to push

Definition: Nondeterministic PDA

A pushdown automaton is a septuple (7-tuple), M (Q, Σ, Γ, q0, ◁, A, δ), where

  • Q is a finite set of states;

  • Σ is a finite set of input symbols;

  • Γ is a finite set of stack symbols;

  • q0 ∈ Q, where q0 is the start state;

  • ◁ ∈ Γ, where ◁ is the starting symbol for the stack;

  • A ⊆ Q, where A is the set of accepting states;

  • δ is a transition function. \[ δ：Q×(Σ∪{Λ})×Γ→2^{Q{×Γ^*}} \]

Stack actions

PDAs have 4 kinds of stack actions

POP (r, Λ) ∈ δ (q, a, g) PUSHing nothing (the string Λ) leaves the automatic POP as the only change
no change (r, g) ∈ δ(q, a, 9) PUSHing the symbol that was on top of the stack, g, undoes the automatic POP, leaving the stack unchanged.
PUSH (r, xg) ∈ δ(q, a, g), with x ≠ Λ. PUSHing a string that ends in g means that g goes onto the stack first, undoing the effect of the automatic POP, so that the result is purely a PUSH of the string x.
POP; PUSH (r, x) ∈ δ(q, a, g), with x ≠ Λ and x not ending in g。 This is a POP of g followed by a PUSH of x.

Parsing

Top-down paraing

  • Parsing consists in finding a derivation tree for an input string and a grammar

  • E.g. String aabb using the following grammar (L(G) = {anbn}):

    ({S}, {a,b}, S, {S → a S b, S → Λ})

  • Start with the root of the tree (start symbol of grammar)

  • Expand the tree downwards using grammar rules

  • Stop when the input string is completely covered

Parse tree

Apply S → aSb; S → aSb; S → Λ

Vague parsing strategy (nondeterminism when expanding S)

NPDA for top-down parsing

  • Grammar: G = ({S}, {a,b}, S, {S → a S b, S → Λ})

  • NPDA M accpets all the strings of L(G) and no others, where

    M = ({q0, q1, q2}, {a, b, S, ◁}, {a, b}, q0, ◁, {q2}, δ)

  • δ:

  • # type from input stack changes
    1 Start q0 Λ ◁ {(q1,S◁)}
    2 Expand q1 Λ S {(q1, a S b), (q1, Λ)}
    3 Match q1 a a {(q1, Λ)}
    4 Match q1 b b {(q1, Λ)}
    5 Stop q1 Λ ◁ {(q2, ◁)}

Different types of rules

  • Start
    • Push S
  • Stop
    • All input symbols consumed + empty stack: stop
  • Expand
    • If the symbol currently at the top of the stack is a nonterminal, non-deterministically replace that symbol by each of its possible right-hand sides
  • Match
    • If the same terminal appears as both the current symbol of input and the top of the stack, it means that the prediction has come true. Pop from stack and advance past this input

Conclusions

  • Context free languages are languages that can be generated by context free grammars
  • Context free languages can also be generated by push down automata
  • Push down automata are usually non-deterministic, in general we cannot make those automata deterministic.
  • We always try to define an unambiguous grammar to make sure the semantics is well defined.
  • How to prove a grammar is unambiguous, is out of scope of this course
