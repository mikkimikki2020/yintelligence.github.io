Concurrent Processes

Video + Slide

在本章中，将介绍与多道程序设计相关的一些方面，即准同时 quasi-simultaneous执行execution和交错interlacing多个程序或程序部分。

这些主题很重要，因为它们可以出现在工程师生活的许多方面，并且因为非 IT 工程师并不总是意识到其中包含的陷阱。

Processes and Threads

Process

所谓的“进程process”是在其执行期间的顺序程序sequential program或顺序程序部分。 这里强调的是进程的顺序性质，这意味着特定process的所有指令都按照指令的顺序和数据值的完全确定的顺序执行。

在任何多道程序设计系统multiprogramming中，多个进程交替执行。同时运行的进程process数由计算机可用的处理器processor数量决定，每个处理器可以保证一个进程的活动。

许多程序由多个必须同时执行的进程组成。 例如，在图形界面中，每个按钮都是一个独立的进程，等待用户选择它。 发生这种情况时，它会向负责执行所选命令的进程发送一条消息。

  • Sequential part of program while it is execute
    • Actions performed entirely determined by
      • Process code
      • Input data
  • In a modern operating system , execution of several processes is interleaved in time.
  • As many processes simultaneously active as processors
  • One program can be composed of several processes
    • Example : in a GUI every button is a process waiting to be selected by the user

Active Process Switch

当进程管理器process manager决定active进程 P0 必须将其状态让给进程 P1 时会发生什么？

  • 首先，借助中断，停止 P0 的执行，并将其状态 state（该进程使用的所有变量的值）保存在为此目的保留的内存部分 PCB0 中。

  • 然后 P1 的state从 PCB1 重新加载。

    当 P1 不再是活动进程时，该状态被保存。

  • 最后，进程管理器process manager通过跳转jump指令激活进程P1。

如果稍后 P1 必须再次将处理器让给 P0，则必须执行逆操作。

在保存进程状态并重新加载另一个进程状态时，计算机不提供“生产性”工作。 这种损失的时间实际上是多道程序的主要成本。

进程的状态state由属于该进程的所有变量的值以及管理系统用于管理该进程的所有表的内容组成，特别是该进程特有的段表segment table和页page。

正如传统上我们所说的“状态”所有变量的值一样，我们经常使用术语“上下文context”来表示在活动进程更改期间要存储的所有数据的集合。

段表segment table和页page的大小可能相当大，改变活动进程的时间成本可能很大，这将严重反对软件工程中越来越多地分解大量进程中的每个程序的趋势.

但是，可以观察到，当多个进程属于同一个程序时，保护每个进程自己的内存空间不受其同事的活动影响并不是必需的。因此，为每个程序定义了一个特定的内存空间，其中包含段表segment table和页page，构成程序的所有进程都可以访问这些表。

这创建了两种类型的进程，自然进程natural process，受到良好保护，具有大上下文和轻进程light process，要保存的状态很小，因为它们是大型进程的“占用squatting”内存空间，这种轻量级进程称为“线程thread”。

Concurrent & shared variables

多个进程在同一地点的合作可能是敏感问题的根源，这些问题将在本章中通过示例进行介绍。

Critical sections

问题是两个进程 P1 和 P2 共享同一个变量 v 。 它们都可以更改其值，而另一个将旧值保留在其寄存器register之一中。

为了避免这种错误，我们应该能够确保当一个进程正在使用变量 v 时，另一个进程不能以任何方式访问 v。为了实现这一点，可以在操作系统保护的每个进程中定义一个临界区critical section，以确保在每一时刻只有一个进程可以进入其critical section。

在示例中，来自 P1 的指令 v = v + 1 和指令write(v); 并且来自 P2 的指令 v = 0 应该受critical section保护。

Software implementations

Example: railroad crossing

要理解临界区，使用铁路类比是很有用的。

考虑轨道交叉。 很明显，没有两列火车可以同时进入交叉路口，这类似于临界区。 红色和绿色信号量控制交叉区域的入口，以防止碰撞。

我们必须找到一种算法，确保不会有两列 2 列火车同时访问临界区，而不会造成不必要的延迟。

With Boolean reservation

想到的第一个想法是基于系统的完美对称性，并使用耦合coupled灯，如果交叉路口空闲，则显示相同颜色的绿色，否则显示红色。

一列火车到达入口点的交叉口会发出红色信号，通过过程的结束确保交付到绿色。

这个系统是危险的（也是真正的铁路灾难的基础），因为虽然不太可能，但永远不能排除两列火车将同时出现在临界区的入口处。它们会导致两个信号灯都变红，而且两列火车都进入了临界区。

在计算机过程中实现这个（坏）想法很简单。 只需对所有需要保护的进程使用公共Boolean变量即可。 这就是下图中显示的自由变量。

...
do {} while (free!);
free = false;
/* critical section */
... ... ...

这个算法的风险在于指令do {} while (free!);和free = false;。 此时，进程知道它可以进入其临界区，但尚未阻止其他进程对其临界区的访问。 多个进程同时处于其临界区的风险是真实的，这是对临界区原则的否定。

With run indicator

为防止两列火车同时进入交叉路口，只需安排两个信号同时为绿色即可。 只需交叉接线即可。当列车 p 到达且 p 亮绿灯时，他进入交叉路口，并在通过终点时引起信号变化。 因此，Q 列车可以轮到他通过十字路口。

过境安全有保障，但连续两列列车不能通过同一通道。 第二个将不得不等到火车通过另一条路径。

enum trun {pturn, qturn};
... ... ...
do {} while (turn !+ pturn);
	/* critical section */
turn = qturn;
... ... ...

前面算法程序中的翻译很简单。 只需创建一个枚举enumeration类型的共享shared变量 turn，它可以采用两个值 pturn 和 qturn。

在进程P中插入的代码只是等到turn有pturn值才能进入临界区，不要忘记在临界区完成时将turn赋值给qturn值。在Q过程中插入的代码是一样的。

再次保证了安全性，但强加了 P 和 Q 过程之间的严格交替，这通常是不可接受的。

With double indicator

为了避免上述算法强加的严格交替，可以使用稍微复杂一些的算法：

到达路口后，每列火车通过分别为布尔变量 pneed 和 qneed 分配true 来宣布它的到达。

出发时，每列火车将其关联变量重置为 false。

两个信号通常都是红色的，但如果 pneed 和 qneed 的值显示交叉操作的合意性和安全性，则每个信号都可以是绿色的。

然而，仍然存在一个主要问题：如果 2 列火车同时到达，pneed 和 qneed 变量都为真，则灯将保持红色并且火车永远停留在十字路口。

双重预订的编程与之前的算法一样容易，只需使用两个布尔变量 pneed 和 qneed。 添加到进程 P 以控制对其临界区的访问的代码显示如下：

boolean pneed, qneed;
... ... ...
pneed = true;
do {} while (turn != pturn);
	/* critical section */
pneed = false;
... ... ...

就像火车一样，进程 P 和 Q 可能最终会在试图无限期地运行它们的指令do while (!qneed)和do while (!pneed)时卡住。

Deckers algorithm

事实上，只有结合每轮技术和双重保留的技术，我们才能找到一个令人满意的算法，通过软件实现一个可靠的临界区，没有阻塞风险。

只有在双重预订导致阻塞的情况下才使用：我们通过交替偏爱 P 或 Q 来摆脱这种阻塞。这种组合以其发明者荷兰计算机科学家德克斯的名字命名。

以下显示了要包含在流程 P 中的代码。 Q 流程的代码是相同的，做必要的修改。

boolean pneed, qneed;
enum trun {pturn, qturn};
...
pneed = true;
while (qneed) do
{
    if (turn == qturn)
    { 
        pneed = false;
        do {} while (turn != pturn);
        pneed = true;
    }
}
	/* critical section */
pneed = false; turn = qturn;
...

Deckers 算法是临界区的软件实现。甚至可以正式证明这种实现是正确的。 不幸的是，Deckers 算法有一个很大的缺点，等待过程继续消耗 CPU 时间。

为了证明，只需查看指令do {} while (turn != pturn);因此，它在实践中很少使用。

Hardware implementations

由于临界区的软件实现导致资源的浪费，因此提供了硬件机制来实现临界区。

在具有单个处理器的系统中，active进程抑制中断inhibit interrupts以创建临界区就足够了，因为另一个进程只能通过interrupt激活。如果使用这种技术，人们不能忽视直接内存访问控制器direct memory access controller也是一种可以改变共享变量值的处理器活动。

幸运的是，这样一个控制器的操作很简单，我们可以很容易地防止他的干扰。

对于多处理器系统，禁止中断不足以创建临界区。在这样的系统中，确实有许多同时活动的进程，因为有处理器。如果我们要避免 Deckers 算法，这将需要特定的硬件设备来实现临界区。

Test & Set bit

测试和设置位test & set bit是一个布尔变量Boolean variable，它的值可以通过读取来改变！

如果变量为 false，则读取操作不会更改此值。 但是，如果为true，则读取操作将返回此值，同时将变量的值从true更改为false。

如果我们重新考虑使用简单保留布尔值来尝试实现临界区的错误算法，但通过使用test & set bit实现布尔值free，我们会看到获得进入critical section的授权并阻止进入任何 现在可以在一个不可分割的事务中完成其他过程。因此，没有 2 个进程可以同时观察critical section的可用性并进行他们认为是排他的保留。

Semaphores

诸如用于制作critical section的那些机制是微妙的。

因此，重要的是不要让应用程序程序员小心地实现它们，而是为此目的将它们包含在特定的功能管理系统中。然后应用程序可以调用这些函数来创建临界区，而用户不必知道其实现的细节。该领域最常用的工具是“信号量semaphore”。

Variable s exclusively accessed through two atomic functions, wait & signal.

void wait( int &s )
{ //  explanaion with busy wait
    do {} while {s =< 0};
    s = s - 1; 
} // wait

void signal( int &s )
{ //  explanaion with busy wait
    s = s + 1; 
} // signal

以上显示了操作称为信号量的共享变量的 2 个函数的语义。

  • 函数wait(&s)等待信号量变为正值，然后将其值减一。
  • 函数signal(& s)增加信号量值。

可以看到等待函数包含一个等待循环do {} while (s = <0);从中可以得出结论，此功能，如 Deckers 算法必须在等待期间保持活动状态。

但是情况并非如此，此处显示的等待实现仅旨在解释此功能，而不是它是如何执行的。

Critical sections with semaphore

/* Declare a semaphore varibale mutex */
/* coomon to all processes */
/* with an initial value of 1 */
int mutex = 1;

...
wait(mutex);
	/* critical section */
signal(mutex);
...

上面的代码显示了如何使用信号量semaphore实现临界区。

信号量semaphore，mutex初始化为值 1，在将使用临界区的不同进程之间共享。

在进入其临界区之前，每个进程都必须调用wait(& mutex)函数。 此函数将阻止相关进程，因为mutex具有负值或零值。 离开临界区，进程必须调用signal (& mutex)函数来解锁另一个阻塞在等待函数中的进程。

Resource allocation with semaphore

信号量semaphore甚至可以概括临界区的概念：例如，如果一台计算机随时有 10 个 CD 刻录机，则不能有 10 个以上的 CD 刻录进程处于活动状态。

可以使用与用于临界区的技术完全相同的技术来评估这种情况，但将信号量设置为值 10 而不是 1。

Semaphore implementation

Busy waiting avoided by using the process scheduler

void wait (int&s)
{
    s = s - 1;
    if ( s < 0 ){
        Add this process to list of processes bloecked by s;
        Block this press;
    }
} // wait

为了在不涉及活动等待资源浪费的情况下实现wait和signal函数来操作信号量，操作系统的进程管理中包含了函数。

如果在函数wait(&s)中，信号量s的值减1后，看起来变为负数，调用wait的进程被pending，并加入到与s关联的pending进程列表中。

一个进程可以有三种状态：

  1. “pending”：进程在内存中，但正在执行其他进程的指令；
  2. “active”：正在执行进程的代码；
  3. “blocked”：由于一些外部原因，例如输入/输出操作，进程无法继续。

活动active状态和阻塞blocked状态之间的转换是当信号量 s 为零或负值时调用函数 wait(&s) 的结果。

通常这种转换是作为输入-输出操作启动的结果出现的，但实际上很多其他原因可以阻塞一个进程，输入输出操作像所有其他潜在的阻塞原因一样使用信号量机制来确保所需的同步 .

void signal (int&s)
{
    s = s + 1;
    if ( s <= 0 ){
        remove a process P from list of processes blocked by s;
        Wake up process P;
    }
} // signal

函数signal(&s)的实现也是进程管理器的一部分。

此外，增加 s 的值，它会检查阻塞进程列表中是否存在可以在增加 s 值后解除阻塞的进程。

进程P1执行信号函数在进程P1的状态图中没有可见的影响。

实际上，这种执行只能在 P1 处于活动状态并且信号对运行它的进程的状态没有影响时发生。 缺点是，如果进程 P2 被wait(&s)的执行阻塞，则 P1 执行signal(& s) 可能会对进程 P2 的状态产生影响。 在这种情况下，signal(&s) 的执行可能具有将 P2 的状态从“blocked”更改为“pending”的效果。

如果有多个进程挂起，信号函数的执行显然只能liberate一个。释放进程的选择在信号量的定义中没有具体说明，是每个操作系统的特定选项。 一般情况下，阻塞时间最长的进程会被释放，但可以实施复杂的优先级规则。

Deadlocks

使用并发进程时可能出现的最后一个重要问题是不可避免的相互阻塞mutual blocking。

在日常生活中，致命阻塞的情况并不少见，甚至经常有警告声明，如果照章办事，将不可避免地导致这种阻塞。因此，在 19 世纪后期，俄亥俄州立法者希望在属于竞争公司的 2 条铁路线交叉点处大力防止任何碰撞风险。 为此，他们颁布了一项法律，规定任何火车将在过境点前一英里处停下，并且在另一列火车先行之前不能离开。

"All trains shall stop at one mile from the crossing and wait until other trains are out of sight." - Ohio State Law, 1896

Evolution of processes

为了能够理解相互阻塞机制，使用一个图表来绘制可能相互挂起的进程的演变是很有用的。

这样的图表具有与考虑的过程一样多的维度。出于易于理解的实际原因，上图仅限于表示 2 个进程的状态。每个轴从 0%（早期(early stage）到 100%（完成finish）。

在一个处理器的系统中，不同的进程交替工作，代表两个被考虑进程的演变的图表将具有阶梯形状，因为每次只能有一个进程在进行。

在多处理器系统的情况下，状态将由在代表 2 个进程结束的点处连接原点（start of the 2 processes）的连续线表示。

在下一节中，将只使用实线，因为相互阻塞的研究没有必要区分单处理器和多处理器。相互阻塞可能有多种原因。为简单起见，这里只考虑资源共享造成的阻塞。其他阻塞原因很容易减少到这一点。我们将通过与轴平行的彩色线显示每个进程在其执行的哪个阶段需要哪些资源。

Deadlock

进程 P1 和 P2 都使用部分执行资源 R1 和 R2。

使用 R1 和 R2 分别由平行于轴的红线和蓝线表示。代表了两条不同的路线，第一个绿色不会导致阻塞blockage，而第二个紫色导致死锁deadlock。

在绿色路线中，P1 一开始进展很快，在 P2 请求之前获得了资源 R1和 R2。

当 P2 需要资源时，它无法获取，一直阻塞，直到 P1释放这些资源。然后 P1和 P2 可以毫无问题地达到它们的最终状态。

在紫色路线中，P2首先询问R1，R1收到它。然后 P1请求 R2，它也收到它。稍后，P1需要 R1，而 R1已分配给 P2。

P1被阻塞，直到 P2 释放 R1，但在释放 R1之前，P2 请求 R2。 P2 也将被阻塞，直到 P1 释放 R2。因此，进程 P1 和 P2 都被阻塞，并且将无限期地保留，因为它们相互依赖才能被释放。

在图中可以看到，从紫色轨迹进入红色矩形的那一刻起，也就是说，当P2和P1收到R1 R2时，情况就无可救药地妥协了。两个进程仍然可以在红色矩形内继续，但无法退出。红色矩形称为“死锁区deadlock zone”。

Deadlock handling

What can be done to avoid a deadlock?

有两种可能的策略：预防prevention和纠正correction。

为了理解这些不同的策略，我们可以将死锁区deadlock area比作一个人可能掉进的洞。

  • 防止堵塞prevent blockage可以是堵塞孔或在周围安装围栏。
  • 修正correction只是在一个人掉到那里后提供了一个从洞里出来的梯子。

下一节将展示如何通过对共享资源的请求施加特定顺序来将阻塞区域减少到零，以及如何通过对所有分配的资源执行一些尽职调查来避免进入阻塞区域。最后，还将描述允许摆脱僵局情况的技术。

  • Prevention
    • Eliminate Deadlock Area
      • Request all resources together
      • Request resources in fixed order
    • Put fence around Deadlock Area
      • Bankers Algorithm
  • Deadlock Correction
    • Provide escape from Deadlock

Deadlock prevention

Request all resources together

在双进程系统的情况下，阻塞区blocking area是对应于两个进程都已经拥有追求另一个进程所必需的资源的状态的区域。

通过强制所有进程同时请求他们需要的所有资源，阻塞区域blocking area的面积减少到零，不再有任何阻塞。

这种技术的唯一缺点通常是过早地将资源预留给其他进程。

Request resources in fixed order

消除阻塞区的一种方法比之前的方法稍逊一筹，是对所有资源进行编号，并强制所有进程按编号递增的顺序获取资源。

因此，在上一节中使用的示例中，P1 应该在单个请求中询问 R1 和 R2，因为它在 R1 之前需要 R2，而 P2 可以从询问 R1 开始，并且只询问真正需要的 R2。

与前一种技术一样，这种技术减少了零阻塞区域zero blocking area，并且在不必要的保留资源方面可能会稍微便宜一些。

Bankers algorithm
  • All processes needing more than one resource have to declare in advance ask resources they will request
  • Time needed to find all loops grow exponentially with the number of nodes !

由 Nico Haberman 发明的被称为“Bankers”的技术不是试图去除阻塞区，而是通过阻塞任何需要分配资源的进程进入阻塞区来阻止进入系统。

为了实现这个算法，每个进程必须提前宣布它需要的所有资源（如果需要一个以上的资源，因为一个进程只使用一种资源永远不会导致阻塞）。

阻塞预防系统维护一个表格，其中列出了每个进程可能询问的所有资源以及一个图表，其节点都是已知的系统进程。

每当一个进程请求资源时，预防系统通过添加一个箭头来完成该图，该箭头从申请进程指向所有其他可能请求相同资源的进程，并且如果新图不包含循环则不授予该资源。

该算法的原理将在下一节中说明，并以一个使用两个以上进程和两个以上资源的示例为例，因为一个具有 2 个进程和 2 个资源的示例无法展示算法的本质。

在上图中显示的示例中，P1 请求 R1 并获得满意，因为图中仅包含从 P1 到 P2 的一个箭头。 P2然后询问 R3，添加了从 P2 到 P3 和 P4 的箭头，但由于仍然没有循环 R3 被授予。

最后 P3 询问 R2，图中添加了 P3 到 P1 的箭头，导致出现循环 P1、P2、P3。这意味着如果将 R2 交给 P3，则后者的进程持有 P1 可以请求的资源，P1 已经拥有 P2 可以请求的资源，P2 又持有 P3 可以请求的资源，在其他表示可能导致致命阻塞的情况。

银行家算法在 1970 年和 80 年的大多数多任务操作系统中都使用过，但是图中的搜索循环所需的时间随着该图中节点的数量大致呈指数增长，导致防止阻塞在现代系统中变得非常昂贵数百个进程可以共享系统资源的系统。

Deadlock correction

要摆脱deadlock，您必须首先检测blockage，这可以通过建立与银行家算法中使用的图形相同的图形来完成，但仅限于在特定时间内观察到的过程。

使用相同的图形，它可以确定哪个进程中的哪个资源分配允许进入阻塞区域。

那么要想摆脱deadlock，无非就是将阻塞进程恢复到它获得致命资源之前的状态。

要恢复处于先前状态的进程，您必须保留该进程当时状态的副本。这意味着每个进程的状态在每次资源分配之前都进行了完整的备份，因此成本不可忽略，但往往比预防算法的成本更容易接受，因为无论如何，需要频繁备份系统状态以防止数据丢失在技术事故期间。这意味着deadlock发生在当代系统中，被视为任何其他硬件故障或系统逻辑。
