Input / Output interfaces & Interrupts

Video + Slide

在前面的章节中，已经详细描述了 CPU 的操作。 然而，有一个方面被完全忽略了。 它是计算机与外部世界之间的通信。这将在本章中讨论。

计算机通过称为接口interface的电子设备连接到外围设备peripheral equipment或通信网络。

Memory Mapped Output

在许多计算机（例如 PC）中，存在概念上简单的界面来在屏幕上显示文本或图像。部分主内存具有双重访问权限：

  • 处理器可以像其他内存一样读取和写入该内存
  • 通常称为“图形处理单元 (GPU)”的屏幕控制器也可以访问此内存提取信息以显示在屏幕上。

虽然第一个显示控制器只是将存储在内存中的 ASCII 字符转换为与特定字体相对应的位图，但现代 GPU 是非常强大的专用处理器，可以实时编码或解码视频流，使用适当的阴影旋转 3D 对象并生成所有视觉效果游戏中常见的效果。

现代 GPU 不仅用于控制显示器，还用于加速算法，这些算法可以从图形所需的大规模并行处理能力中受益。一个典型的例子是蛋白质折叠的计算机模拟。

在屏幕上显示文本或图画不需要计算机和屏幕之间的同步机制。这两个组件可以按照自己的节奏工作，而不必担心另一个组件的节奏。

Polled I/O

当计算机和外围设备的节奏必须同步时，使用的接口本质上包含一个数据寄存器（“data”）和一个同步位synchronization bit（“sy”）。

为了说明这些想法，让我们将原始打印机视为外围设备。

  • 当程序需要向打印机发送一个character时，该character被写入接口的 DATA 寄存器并且 Sy 位设置为true。
  • 打印机准备好打印时，会检查 Sy 位的状态。
    • 当 Sy 位为true时，它读取 DATA 寄存器的内容并将 Sy 位设置为false。
    • 程序会定期regularly检查 Sy 位的状态。 当发现 Sy 位为false时，下一个character将加载到 DATA 寄存器中，并且将Sy 位再次设置为true。

Polling Procedure

示例 C 函数使用了接口interface。

void print(char ch);
//Printer polling method
{
    while (Sy) {/* wait */};
    Data = ch;
    Sy = true /* activate pribnter interface */;
}

需要注意的是，在将要打印的字符放入数据寄存器之前，用于等待 Sy 位复位的 while ... do 循环。在循环执行期间，程序只是等待打印机准备就绪，不断验证 Sy 变量的状态。

这种技术已经在本课程第一章描述的电子锁的例子中使用过，叫做“轮询polling”

Interrupt handler

正如在课程的第一部分中已经解释的那样，中断机制interrupt mechanism可用于避免以轮询polling为特征的non-productive waiting activity。

具有interrupt mechanism的计算机的CU有一个或多个输入端。Interrupt terminal的状态改变导致当前程序的中断：在中断期间，控制器完成当前指令的执行，但不是将下一条指令取到该语句确定的地址，而是在另一条指令上寻找它地址由信号中断决定。通常，在调用subroutine时，会保留被中断程序的下一条指令的地址，以便在中断处理后继续执行程序。

由于interrupt而执行的程序通常称为“中断处理程序interrupt handler”。

Interrupt Issues

CU的架构师必须做出两个与中断机制相关的重要选择：

  • Where is the interrupt handler
  • How to save the return address

Location of the interrupt handler

  • Fixed direct address (INTEL)

    每个中断终端关联一个physical address，相应的中断处理程序的第一条指令必须在该physical address上

  • Fixed indirect address (Motorola)

    为每个终端关联一个physical address，指令地址的第一个管理器manager必须位于该physical address

  • Vectored Interrupts (DEC)

    load导致中断的设备，为自己提供第一条指令。

保存中断程序的状态（通常称为“上下文context”）也可以通过多种方式来完成，但应该意识到这还不够，例如在子程序调用的情况下保存返回地址和可能的动态链接，但也应该保存所有可能被中断处理程序改变的寄存器的内容，因为我们无法知道什么时候会发生中断的执行，而且很可能只是当所有寄存器都包含将要执行的中间结果时才发生在程序的其余部分执行期间使用。

Direct interrupt addresses

在某些处理器中，尤其是个人计算机中普遍使用的 X86 系列处理器中，每个中断输入线都与一个固定地址相关联，并且在中断之后，要执行的第一条指令将是在该地址处找到的指令。

  • 在X86处理器中，地址0为RESET interruption，地址4为第一个正常中断，地址8为下一个，依此类推。在复位reset信号 (RESET) 期间，无需在当前程序中保存任何内容。
  • 通常会在地址 0 处找到一条无条件跳转unconditional jump指令，跳转到操作系统的起始地址。

对应于其他中断的地址通常包含对相关中断处理程序的子程序调用。这样，在中断期间，返回地址自动保存在堆栈中，并通过返回指令结束中断处理程序，它自动返回到被中断的程序。对于每个中断处理程序，程序员必须注意备份可能被中断处理程序修改的所有上下文context数据（例如在返回地址上方的堆栈上）。在返回主程序之前，即在返回指令之前，上下文context数据必须恢复到中断前的情况。

Indirect interrupt addresses

在其他处理器中，包括最初由摩托罗拉设计的处理器，每个中断终端都在内存中分配了一个固定地址，控制单元可以在其中获取fetch相应中断处理程序的开头地址。

  • At interrupt, the entire context, including P, is saved on stack by control unit.
  • The RTI instruction restores the entire context, including P

在此类处理器中，控制单元将负责备份back堆栈上所有寄存器的内容。因此，这些处理器具有在恢复所有已保存记录的内容时从中断处理程序返回的特定指令。

这种方式无疑对程序员更加友好，因为他或她不必担心保存被中断程序的context，但是如果该context有相当大的规模，那么对于某些关键应用程序，对中断的响应时间可能会令人望而却步。为此，在某些处理器中，仅自动保存部分context。 在这种情况下，如果在中断执行期间可能会修改其余的context，则由程序员来保护它。

Vectored Interrupts

一些旨在处理来自不同来源的大量中断的处理器没有将地址与特定中断一成不变地关联的机制。

中断后，它们只是开始下一条指令的读取周期，但不寻址内存。

相反，它们赋予引起中断的设备authorization，将其自身置于数据总线上的可执行语句以加载到指令寄存器instruction register中。

the authorization to place itself an executable statement on the data bus to be loaded into the instruction register.

  • After an interrupt, the interrupting device will normally put a JSB instruction on the data bus for calling the interrupt handler

引起中断的设备通常会发送带有所需中断处理程序地址的子程序调用指令。有了这种称为“向量中断vectored interrupt”的机制，很明显程序员对上下文保存负有全部责任。

Save the return address

  • By programmer
  • Automatically, the entire state
  • Automatically, a part of the state
  • By shadow registers

Interrupt Driven I/O

使用中断来管理 I/O 操作使用与polling方法研究中描述的完全相同的接口。

在图中，我们看到 Sy 位直接连接到处理器的一个中断输入。

通过这种方式，程序不应该连续查询 Sy 位的状态，而是可以从事其他活动，甚至可以暂停以允许另一个程序的执行，因为在 Sy 位的状态改变时会发生中断。

该中断允许等待 Sy 位状态改变的程序在 DATA 寄存器中发送以下数据。

void print(char ch);
// Printer fuction pseudocode
{
    while (bufferfull) {/* wait */}
        put ch into buffer;
    if (buffer was empty) {start pdriver}
}

上面函数显示了当打印机接口可以产生中断时如何对打印操作进行编程。 程序中的斜体对应于细节会混淆示例的代码部分。

我们假设存在一个缓冲区buffer，作为单个队列进行管理，“先到先得”。

在正常使用中，打印函数只是将要打印的字符放入缓冲区。该函数还包含一个活动等待循环 while (bufferfull) do {}，但它仅在缓冲区饱和的情况下使用。 如果在调用 print 时缓冲区为空，则打印机处于非活动状态并且不会引起中断 .

这就是为什么通过 if (buffer was empty) {start} pdriver 指令检查缓冲区的初始状态的原因，如有必要，这会导致在发生中断时激活打印机的驱动程序 (pdriver)（它实际上是打印机的中断处理程序）。

void pdriver;
// Interrupt driven printer driver
{
    while (true) 
    {
        if (Buffer is empty) {Suspend pdriver}
        Data = Next character from Buffer;
        Sy = true; (*activate printer interface*)
        Suspend driver until printer interrupt
    }
}

由打印机引起的中断由称为“打印机驱动程序printer driver”的特定程序管理。

驱动程序包含一个无限循环 while (true) do 在此循环中重复执行两个操作：

  • 首先，从缓冲区中删除第一个字符并发送到接口的 DATA 寄存器，

  • 然后驱动程序处于阻塞blocked状态state，直到打印机引起中断。

如果在驱动程序激活时缓冲区为空，则驱动程序立即被阻塞，不希望通过中断释放。

Performance of interrupt Driven I/O

  • Some orders of magnitude :
    • Processor speed = 50 Millions of Instructions Per Second
    • One loop of the driver = 50 instructions
    • Duration of one IO operation = 1 HS
    • Maximum I / O throughput = 1 Mbytes / S
    • Data rate on a 100 Mb / s Ethernet = 12 . 5 Mbytes / S
    • Data rate on a 48 程 D reader = 10 Mbytes / s
  • Conclusion
    • Interrupt driven I / O much too slow for fast peripherals and local area networks .

要估计使用中断机制可实现的 I/O 速率，可以考虑以下magnitude：

  • 每秒执行 5000 万条指令的处理器
  • 在其主loop中包含 50 条指令的驱动程序

这导致传输一个字节大约需要一微秒，并且最大可达到的流速约为a megabyte per second (1 MB/s)。

由于本地网络的其他吞吐量大约为每秒 1 到 20 兆字节，并且在硬盘驱动器和 CD 驱动器中发现相同数量的吞吐量，人们意识到中断管理的操作 I/O 设备太慢了。

由中断直接管理的慢速 I/O 操作直接影响 CPU 的通用性。 由于 CPU 可以执行多种任务，因此需要 50 条指令才能准确描述在内存和 I/O 接口之间简单传输一个字节的任务。

Direct Memory Access Controller

为了加速 I/O，在计算机中添加了一个专门的处理器。该处理器只能完成一项任务，包括在内存和 I/O 接口之间传输字节。

只需一个简单的控制信号就足以进行传输，而无需从内存中读取任何指令。这种专用处理器称为“直接内存访问控制器Direct Memory Access Controller”。

在传输之前，CPU 执行的程序通过在 DMAC 中加载源地址、目标地址和要传输的字节数来准备传输。

接口的 Sy 位连接到 DMAC 的传输控制。每当 Sy 状态指示接口可用时，DMAC 就会传输下一个字节。当指定块block的所有字节都传输完毕后，DMAC 通过引起中断通知请求程序。所以我们可以说 DMAC 将中断的数量除以传输数据的块block大小。

由于 DMAC 可以传输一个字节byte，因此它必须对地址和数据总线具有权限。

为了在每次传输之前获得此权限，DMAC 通过总线向 CPU 请求权限。后者然后结束当前的memory cycle，将总线传输到 DMAC 并避免使用总线，直到 DMAC 完成字节的传输。他们说 DMAC 会从 CPU 中窃取memory cycle，因此会稍微减慢程序的执行速度。

一个DMA Controller必须至少有四个寄存器register：

  1. 第一个包含必须读取字节传输的地址（“源地址Origin Address”）。
  2. 第二个包含必须写入该字节的地址（“Destination 地址”）。
  3. 第三个包含要传输的字节数（“字节计数Byte Count”）。
  4. 第四个用于为特定传输配置DMAC。

此配置决定了在每个字节传输中必须如何调整各种寄存器的内容。

例如，当需要将数据块从主内存复制到磁盘控制器disk controller时，目标地址保持不变，而随着每个字节的传输，源地址Origin Address增加，字节计数Byte Count减一。
