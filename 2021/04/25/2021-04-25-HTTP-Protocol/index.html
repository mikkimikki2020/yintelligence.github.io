HTTP Protocol

Background + Question + Slide + Video

HTTP Request

An HTTP Request is sent to the server by the client (browser).

  • GET

    (’GET’ information from ...)

  • POST

    (’POST’ information to ...)

  • PUT

    (’PUT’ information inside ...)

  • DELETE

    (’DELETE’ something from ...)

  • OPTIONS

    (What are my ’OPTIONS’ with ...)

  • HEAD

    (Only fetch the headers, used to determine file sizes)

  • TRACE

    (Debugging request, Apache recommends it being disabled)

Note: Regular browsing usually only uses GET and POST requests.

An HTTP Request contains information about the client/browser represented as ’HTTP Request Headers’.

  • Browser Information (User-Agent):Type of browser,version, platform, ...
  • User Information (Cookies, Authorization):Session information, settings, ...
  • Request Data: Form data, parameters, ...

Note: Some things can be inferred by the server, such as yourIP-address,... (even when it is not included in a request)

Example to https://www.vub.be/en:

GET https://www.vub.be/en HTTP/1.1
Connection: keep-alive
Cache-Control: max-age=0
Upgrade-Insecure-Requests: 1
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)... Chrome/77.0.3865.90 Safari/537.36↪→
Sec-Fetch-Mode: navigate
Sec-Fetch-User: ?1
Accept: text/html,application/xhtml+xml, ...
Sec-Fetch-Site: same-origin
Referer: https://www.vub.be/
Accept-Language: nl-BE,nl-NL;q=0.9,nl;q=0.8,en-US;en;
Cookie: vubcookie=2; s_pers=...; has_js=1
Host: www.vub.be

HTTP Response

An HTTP Response is the information that is returned after posting an HTTP Request. It contains:

  • Response Status: Represents the status of the request. There are many different status codes available.
  • Response headers: Server info, policies, cookies, content info (type, length) ,...
  • Response Body: The content that is returned.

HTTP Proxy

  • Forwards requests to the actual website
  • Forwards responses to the client that made the request

Cookies

  • HTTP is stateless!This means that it does remember who or what is browsing between HTTP requests (e.g. page clicks)
    • Solution: Cookies
  • Browser-kept data for sites.
  • Used for user sessions, language settings, ...
  • Browser needs to provide the cookies with each request to the site that requires it.
  • Uniquely identify users with a code
  • Randomized with letters and numbers (You don’t want hackers to guess the codes)
  • Sessions prevent you from having to log in again when changing pages
  • Cookies can be transferred to another browser
    • ”Cookie stealing”
    • Extra security measures such as checking the IP, checking the User-Agent,

Exercise

In these exercise session you will have to complete several challenges. Each challenge tests your knowledge on the HTTP Protocol.

Some of these challenges require you to perform actions that are not possible in a normal browser. The requirements below will show you how to use OWASP Zed Attack Proxy. As mentioned in the lecture a proxy is situated between the client (your browser) and server. The proxy that we will use allows you to intercept and edit HTTP Requests before being forwarded to the server.

You can find the questions here.

Requirements

  • OWASP ZAP (Cross-Platform Package). This allows you to intercept and edit HTTP Re-quests before they are sent to the server.
  • Browser add-on for turning the proxy on or off. Proxy SwitchyOmega for Chrome or Firefox.

Note: Google Chrome is recommended. Instructions and tutorial screenshots will be made within Chrome.

Configuration

  1. Start OWASP ZAP by clicking thezap.bat(Windows) or executing thezap.sh(Linux/macOS).

  2. When asked if you want to persist sessions, select ”No”

  3. Go to ”Tools”<”Options”

  4. Select ”Local Proxies” on the left side, and change the port number (8080 by default) to 8082to make sure it does not conflict with other services.

  5. Open the SwitchyOmega configuration in your browser (Click here for Chrome)

  6. Create a new Proxy Profile in SwitchyOmega called ”ZAP”

  7. Configure the proxy as ”HTTP” with hostname: ”localhost” and port ”8082”.

    Note: When you want to use this tool for a locally hosted website, make sure to clear the blacklist

Usage

Chrome - Development Console

Many browsers have a development console. This console allows you to see HTTP requests and responses (to pages, images and other resources), JavaScript console, dynamic DOM manipulation(live editing of the HTML pages) and the loaded cookies.

  1. Press F12 to open the Development Console2. The first tab is called ”Elements”.
  2. Here you can view and manipulate HTML DOM elements.
  3. The second tab is called ”Console”. Here you can view the JavaScript logs (errors, ...) and execute your own JS code.
  4. The fourth tab is called ”Network”. Here you can view the HTTP requests and responses. However, this is only a history view - you can not edit the requests. This tab can be useful to determine how long it takes for each element of your page to load.
  5. Finally, the ”Application” tab contains persistent data about the website. We are mainly interested in the cookies. You can add, edit or remove cookies that are included in HTTP requests.

OWASP ZAP - Editing HTTP Requests

  1. Start OWASP ZAP by clicking thezap.bat(Windows) or executing thezap.sh(Linux/macOS).
  2. When asked if you want to persist sessions, select ”No”
  3. Open your browser. You should see an icon on the top of the installed proxy plugin (Switchy-Omega)
  4. Select the ZAP profile you have created.
  5. You may see a message saying ”Your connection is not private”. Your browser warns you ofa possible MITM (Man In The Middle) attack, in this case that is exactly what we are doing.
  6. Go back to OWASP ZED, you will see the GET HTTP Request. From now on, every requestyou make will show up in OWASP ZED.
  7. In order to ”Edit” requests you will have to insert a breakpoint.
  8. When you make a request, you will see a new tab in ZAP where you can edit the request.
