Uniform Resource Identifier 统一资源标识符 (URI)

A Uniform Resource Identifier (URI) uniquely identifies a resource

统一资源标识符URI唯一地标识一个资源

There are two types of URIs

  1. Uniform Resource Locator 统一资源定位符 (URL)

    • contains information about the exact location of a resource

      包含有关资源确切位置的信息

    • consists of a scheme, a host and the path (resource name)

      由方案 scheme、主机host 和 路径 path（资源名称）组成

      • e.g. https://vub.academia.edu/BeatSigner

    • problem: the URL changes if resource is moved!

      问题：如果资源被移动，URL 就会改变！

      • Persistent Uniform Resource Locators (PURLs)

  2. Uniform Resource Name 统一资源名称 (URN)

    • unique and location independent name for a resource

      资源的唯一且与位置无关的名称

    • consists of a scheme name, a namespace identifier and a namespace-specific string (separated by colons)

      由方案名称 scheme name、命名空间标识符namespace identifier 和 特定于命名空间的字符串（以冒号分隔）组成

      1. e.g. urn: ISBN:3837027139

HTTP Protocol

  • Request/response communication model

    请求/响应通信模型

    • HTTP Request
    • HTTP Response

  • Communication always has to be initiated by the client

    通信始终必须由客户端发起

  • Stateless protocol (no sessions)

    无状态协议（无会话）

  • HTTP can be used on top of various reliable protocols

    HTTP 可用于各种可靠协议之上

    • TCP is by far the most commonly used one

    • runs on TCP port 80 by default

      默认在 TCP 端口 80 上运行

  • HTTPS scheme used for encrypted connections

    用于加密连接的 HTTPS 方案

HTTP Message Format 消息格式

HTTP_message = start_line , {header} , "CRLF" , {body};

HTTP Request Message 请求消息

Request-specific start line 请求特定的起始行

start_line = method, " " , resource , " " , version;
method = "GET" , "HEAD" , "POST" , "PUT" , "TRACE" , "OPTIONS" , "DELETE";
resource = complete_URL | path;
version = "HTTP/" , major_version, "." , minor_version;

Methods

  • GET : get a resource from the server 从服务器获取资源
  • HEAD : get the header only (no body) 仅获取标题（无正文）
  • POST : send data (in the body) to the server 将数据（在正文中）发送到服务器
  • PUT : store request body on server 在服务器上存储请求体
  • TRACE : get the "final" request 获取“最终”请求 (after it has potentially been modified by proxies 在它可能被代理修改之后)
  • OPTIONS : get a list of methods supported by the server 获取服务器支持的方法列表
  • DELETE: delete a resource on the server 删除服务器上的资源

HTTP Response Message 响应消息

Response-specific start line

start_line = version , status_code , reason;
version = "HTTP/" , major_version, "." , minor_version;
status_code = digit , digit , digit;
reason = string_phrase;

Status codes 状态码

  • 100-199 : informational
  • 200-299 : success (e.g. 200 for 'OK')
  • 300-399 : redirection
  • 400-499 : client error (e.g. 404 for 'Not Found')
  • 500-599 : server error (e.g. 503 for 'Service Unavailable')

HTTP Header Fields 标头字段

There exist

  • general headers 通用标头 (for requests and responses)
  • request headers 请求标头
  • response headers 响应标头
  • entity headers 实体标头
  • extension headers 扩展标头

Some important headers

  • Accept

    • request header defining the Media Type (formerly MIME type) that the client will accept

      请求标头定义客户端将接受的媒体类型（以前的 MIME 类型）

  • User-Agent

    • request header specifying the type of client

      指定客户端类型的请求头

  • Keep-Alive (HTTP/1.0) and Persistent (HTTP/1.1)

    • general header helping to improve the performance since otherwise a new HTTP connection has to be established for every single webpage element

      通用标头有助于提高性能，否则必须为每个网页元素建立新的 HTTP 连接

  • Content-Type

    • entity header specifying the body's MIME type

      指定正文 MIME 类型的实体标头

  • If-Modified-Since

    • request header that is used in combination with a GET request (conditional GET)

      与GET请求（条件 GET）结合使用的请求标头

    • the resource is only returned if it has been modified since the specified date

      仅当自指定日期以来已修改资源时才返回资源

媒体类型定义请求或响应正文的内容（用于适当的处理）

Proxy Caches 代理缓存

A proxy cache is a special type of proxy server

代理缓存是一种特殊类型的代理服务器

  • can reduce server load if multiple clients share the same cache

    如果多个客户端共享相同的缓存，可以减少服务器负载

  • often multi-level hierarchies of caches (e.g. continent, country and regional level) with communication between sibling and parent caches as defined by the Internet Cache Protocol (ICP)

    通常是缓存的多级层次结构（例如大陆、国家和区域级别），并在Internet 缓存协议 (ICP) 定义的兄弟缓存和父缓存之间进行通信

  • passive or active (prefetching) caches

    被动或主动（预取）缓存

Special HTTP cache control header fields 缓存控制头字段

  • Expires

    • expiration date after which the cached resource has to be refetched

      必须重新获取缓存资源的到期日期

  • Cache-Control: max-age

    • maximum age of a document (in seconds) after it has been added to the cache

      文档被添加到缓存后的最大年龄（以秒为单位）

  • Cache-Control: no-cache

    • response cannot be directly served from the cache (has to be revalidated first)

      无法直接从缓存中提供响应（必须先重新验证）

  • ...

Validators 验证器

  • Last-modified time as validator 作为验证者的最后修改时间

    • cache with resource that has been last modified at time t uses an If-Modified-Since t request for updates

      具有上次修改时间 t 的资源的缓存使用If-Modified-Since t请求更新

  • Entity tags 实体标签 (ETag)
    • changed by the publisher if content has changed; If-None-Match etag request
    • 如果内容已更改，则由发布者更改；If-None-Match etag请求

Advantages

  • reduces latency and used network bandwidth

    减少延迟和使用的网络带宽

  • reduces server load (client and reverse proxy caches)

    减少服务器负载（客户端和反向代理缓存）

  • transparent to client and server

    对客户端和服务器透明

Disadvantages

  • additional resources (hardware) required

    需要额外的资源（硬件）

  • might get stale data out of the cache

    可能会从缓存中获取陈旧数据

  • creates additional network traffic if we use an active caching approach (prefetching) but achieve a low cache hit rate

    如果我们使用主动缓存方法（预取）但实现低缓存命中率，则会创建额外的网络流量

  • server loses control (e.g. access statistics) since no longer all requests have to be sent to the server

    服务器失去控制（例如访问统计），因为不再需要将所有请求发送到服务器

Tunnels 隧道

  • Implement one protocol on top of another protocol

    在另一个协议之上实现一个协议

    • e.g. HTTP as a carrier for SSL connections

      例如 HTTP 作为 SSL 连接的载体

  • Often used to "open" a firewall to protocols that would otherwise be blocked

    通常用于向协议“打开”防火墙，否则会被阻止

    • e.g. tunnelling of SSL connections through an open HTTP port

      例如，通过开放的 HTTP 端口建立 SSL 连接的隧道

Gateways 网关

A gateway can act as a kind of "glue" between applications (client) and resources (server)

网关可以充当应用程序（客户端）和资源（服务器）之间的一种“粘合剂”

  • translate between two protocols (e.g. from HTTP to FTP)

    在两个协议之间进行转换（例如从 HTTP 到 FTP）

  • security accelerator (e.g. HTTPS/HTTP on the server side)

    安全加速器（例如服务器端的HTTPS/HTTP）

  • often the gateway and destination server are combined in a single application server (HTTP to server application translator)

    通常网关和目标服务器组合在一个应用程序服务器中（HTTP 到服务器应用程序转换器）

「网络关口」也可以理解成「网络海关」。当你出国的时候，必须经过海关（不考虑偷渡的情况）。但你只在中国大陆旅游的话需要经过海关吗？显然不需要。

如果两台计算机在同一个广播域里，也具有相同的网段，他们之间可以通过 ARP 发现彼此的 MAC 之后直接通讯，并不需要三层的「网关」。

网关某种意义上也具有「代理」的角色。

形象一点说，小明又想送礼物给小美了。小明在 A 国，小美在 B 国。小明并不知道具体去往 B 国小美家的路线，他把礼物包好并写上地址，交给快递员（网关），快递员也不知道具体去往 B 国的路线，他只知道包裹交给 A 国集散中心（另一个网关）就好。A 国集散中心也不知道 B 国小美家具体在哪， 不过 A 国集散中心知道把包裹交给 B 国集散中心（又一个网关），他们会想办法的。包裹送到了 B 国集散中心，B 国集散中心还是不知道小美家具体在哪，交给 B 国快递员（还是一个网关），最后 B 国快递员找到了小美家，把包裹送到了小美手上。

如果不要网关可以吗？理论上也可以，上面的故事中小明也可以自己开车跟着导航把礼物送给小美，但是多麻烦啊。现实中没有网关可以吗？不可能的。想象一下，如果需要访问腾讯，就要从腾讯的机房拉一条网线到家里，想访问网易又从网易拉一条网线。如果每个人都这样拉网线，地球将被网线包围。所以我们交给网关，一跳一跳的把数据送到目的地。

Session Management 会话管理

Session (state) tracking solutions 会话（状态）跟踪解决方案

  • use of IP address

    • problem: IP address is often not uniquely assigned to a single user

      问题：IP地址通常不是唯一分配给单个用户的

  • browser login

    • use of special HTTP authenticate headers

      使用特殊的 HTTP 身份验证标头

    • after a login the browser sends the user information in each request

      登录后，浏览器在每个请求中发送用户信息

  • URL rewriting 网址重写

    • add information to the URL in each request

      在每个请求的 URL 中添加信息

    • similar to URL rewriting but information can also be in body (POST request)

      类似于 URL 重写，但信息也可以在正文中（POST 请求）

  • cookies

    • the server stores a piece of information on the client which is then sent back to the server with each request

      服务器在客户端存储一条信息，然后随着每个请求发送回服务器

Cookies

一文带你看懂cookie，面试前端不用愁

  • Introduced by Netscape in June 1994

  • A cookie is a piece of information that is assigned to a client on their first visit

    cookie 是在客户第一次访问时分配给他们的一条信息

    • list of <key, value> pairs

    • often just a unique identifier

      通常只是一个唯一标识符

    • sent via Set-Cookie HTTP response headers

      通过Set-Cookie HTTP 响应头发送

  • Browser stores the information in a "cookie database" and sends it back every time the same server is accessed

    浏览器将信息存储在“cookie 数据库”中，并在每次访问同一服务器时将其发回

  • Potential privacy issues 潜在的隐私问题

    • third-party websites might use persistent cookies for user tracking

      第三方网站可能会使用持久性 cookie进行用户跟踪

  • Cookies can be disabled in the browser settings

    可以在浏览器设置中禁用 Cookie

cookie是当你浏览某个网站的时候，由web服务器存储在你的机器硬盘上的一个小的文本文件。它其中记录了你的用户名、密码、浏览的网页、停留的时间等等信息。当你再次来到这个网站时，web服务器会先看看有没有它上次留下来的cookie。如果有的话，会读取cookie中的内容，来判断使用者，并送出相应的网页内容，比如在页面显示欢迎你的标语，或者让你不用输入ID、密码就直接登录等等。

当客户端要发送http请求时，浏览器会先检查下是否有对应的cookie。有的话，则自动地添加在request header中的cookie字段。注意，每一次的http请求时，如果有cookie，浏览器都会自动带上cookie发送给服务端。那么把什么数据放到cookie中就很重要了，因为很多数据并不是每次请求都需要发给服务端，毕竟会增加网络开销，浪费带宽。所以对于那设置“每次请求都要携带的信息（最典型的就是身份认证信息）”就特别适合放在cookie中，其他类型的数据就不适合了。

简单的说就是：

  1. cookie是以小的文本文件形式（即纯文本），完全存在于客户端；cookie保存了登录的凭证，有了它，只需要在下次请求时带着cookie发送，就不必再重新输入用户名、密码等重新登录了。
  2. 是设计用来在服务端和客户端进行信息传递的

Server-side processing 服务器端处理

Common Gateway Interface 通用网关接口 (CGI)

CGI was the first server-side processing solution

CGI 是第一个服务器端处理解决方案

  • transparent to the user 对用户透明

  • certain requests (e.g. /account.pl) are forwarded via CGI to a program by creating a new process

    某些请求（例如 /account.pl）通过创建新进程通过 CGI 转发到程序

  • program processes the request and creates an answer with optional HTTP response headers

    程序处理请求并使用可选的 HTTP 响应标头创建答案

CGI Problems

  • a new process has to be started for each request

    必须为每个请求启动一个新进程

  • if the CGI program for example acts as a gateway to a database, a new DB connection has to be established for each request which results in a very poor performance

    例如，如果 CGI 程序充当数据库的网关，则必须为每个请求建立一个新的数据库连接，这会导致性能非常差

FastCGI solves some of the problems by introducing persistent processes and process pools

FastCGI通过引入持久化进程和进程池解决了部分问题

CGI/FastCGI becomes more and more replaced by other technologies (e.g. Java Servlets)

CGI/FastCGI 越来越多地被其他技术（例如 Java Servlets）取代

Java Servlets

A Java servlet is a Java class that has to extend the abstract HTTPServlet class

Java servlet 是必须扩展抽象 HTTPServlet 类的 Java 类

The Java servlet class is loaded by a servlet container and relevant requests (based on a servlet binding) are forwarded to the servlet instance for further processing

Java servlet 类由 servlet 容器加载，相关请求（基于 servlet 绑定）被转发到 servlet 实例进行进一步处理

  • Main HttpServlet methods

    doGet(HttpServletRequest req, HttpServletResponse resp)
doPost(HttpServletRequest req, HttpServletResponse resp)
init(ServletConfig config)
destroy()

  • Servlet life cycle 生命周期

    • a servlet is initialised once via the init() method

      servlet 通过 init() 方法初始化一次

    • the doGet(), doPost() methods may be executed multiple times (by different HTTP requests)

    • doGet(), doPost()方法可能会被多次执行（通过不同的 HTTP 请求）

    • finally the servlet container may unload a servlet (upcall of the destroy() method before that happens)

      最后，servlet 容器可能会卸载一个 servlet（在此之前调用 destroy()方法

  • Servlet container (e.g. Apache Tomcat) either integrated with web server or as standalone component

    与 Web 服务器集成或作为独立组件的 Servlet 容器（例如 Apache Tomcat）

Java Servlet Example

package org.vub.wise;
import java.io.*;
import java.util.Date;
import javax.servlet.http.*;
import javax.servlet.*;

public class HelloWorldServlet extends HttpServlet {
    public void doGet (HttpServletRequest req, HttpServletResponse res)
    	throws ServletException, IOException {
    PrintWriter out = res.getWriter();
    out.println("<html>");
    out.println("<head><title>Hello World</title></head>");
    out.println("<body>The time is " + new Date().toString() + "</body>");
    out.println("</html>");
    out.close();
    }
}

JavaServer Pages (JSP)

  • A "drawback" of Java servlets is that the whole page (e.g. HTML) has to be defined within the servlet

    Java servlet 的一个“缺点”是整个页面（例如 HTML）必须在 servlet 中定义

    • not easy to share tasks between web designer and programmer

      不容易在网页设计师和程序员之间共享任务

  • Add program code through scriptlets and markup to existing HTML pages

    通过scriptlet和markup向现有 HTML 页面添加程序代码

  • These JSP documents are then either interpreted on the fly (Apache Tomcat) or compiled into Java servlets

    然后这些 JSP 文档要么被动态解释（Apache Tomcat）要么被编译成 Java servlet

  • The JSP approach is similar to PHP or Active Server Pages (ASP)

    JSP 方法类似于 PHP 或 Active Server Pages (ASP)

    Note that Java Servlets become more and more an enabling technology (as with JSP)

    请注意，Java Servlet 越来越成为一种支持技术（与 JSP 一样）

Client-side processing 客户端处理

JavaScript

  • Interpreted scripting language for client-side processing

    用于客户端处理的解释性脚本语言

  • JavaScript functionality often embedded in HTML documents but can also be provided in separate files

    JavaScript 功能通常嵌入在 HTML 文档中，但也可以在单独的文件中提供

  • JavaScript often used to

    • validate data 验证数据 (e.g. in a form)

    • dynamically add content to a webpage

      动态地向网页添加内容

    • process events 处理事件 (onLoad, onFocus, etc.)

    • change parts of the original HTML document

      更改原始 HTML 文档的部分内容

    • create cookies 创建 cookie

    • ...

    Note: Java and JavaScript are completely different languages!

JavaScript Example

<html>
    <body>
    <script type="text/javascript">
    	document.write("<h1>Hello World!</h1>");
    </script>
    </body>
</html>

Java Applets

A Java applet is a program delivered to the client side in the form of Java bytecode and runs in a sandbox

Java小程序是以Java字节码的形式交付给客户端并在沙箱中运行的程序

  • executed in the browser using a Java Virtual Machine (JVM)

    使用 Java 虚拟机 (JVM) 在浏览器中执行

  • an applet has to extend the Applet or JApplet class

    小程序必须扩展小程序或 JApplet 类

Java Web Start (JavaWS)

More recently Java Web Start (JavaWS) is replacing Java Applets

最近Java Web Start (JavaWS)正在取代 Java Applet

  • program no longer runs within the browser

    程序不再在浏览器中运行

    • less problematic security restrictions

      问题较少的安全限制

    • less browser compatibility issues

      更少的浏览器兼容性问题

Advantages

  • the user automatically always has the most recent version

    用户自动始终拥有最新版本

  • high security for untrusted applets

    不受信任的小程序的高安全性

  • full Java API available for trusted signed applets

    完整的 Java API 可用于受信任的签名小程序

Disadvantages

  • requires a browser Java plug-in

    需要浏览器 Java 插件

  • only signed applets can get more advanced functionality

    只有签名的小程序才能获得更高级的功能

    • e.g. network connections to other machines than the source machine

      例如，与源机器以外的其他机器的网络连接
